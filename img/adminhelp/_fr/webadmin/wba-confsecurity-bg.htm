Vous pouvez définir des options conçues pour rendre Content Server plus sûr. Pour accéder à la page Configurer les paramètres de sécurité, sur la page Administration Content Server, sélectionnez Configuration du serveur. Cliquez sur Configuration des paramètres de sécurité.

 

Note :
Pour des fonctions de sécurité Content Server supplémentaires, reportez-vous aux rubriques Limitation des connexions du compte administrateur par adresse IP et Suppression des événements en suspens.

 

Cookies HTTP uniquement

La section Cookies HTTP uniquement vous permet de définir si l'attribut httpOnly doit être ajouté à tous les cookies Content Server. Le cas échéant, tous les cookies seront identifiés par l'attribut httpOnly. Dans les navigateurs qui le prennent en charge, cet attribut indique qu'un cookie ne doit pas être rendu disponible pour le script. Par défaut, cette option est désactivée.

 

Note :
L'activation de l'option Cookies HTTP uniquement désactive l'éditeur de workflow et l'éditeur de texte.

 

Clé de codage des cookies

Content Server génère un cookie d'authentification à l'aide de l'ID numérique de l'utilisateur et d'autres valeurs que vous pouvez définir. Les valeurs que vous sélectionnez sont nécessaires pour l'authentification des requêtes après la connexion. La clé de codage des cookies vous permet de générer une chaîne de texte à utiliser comme clé pour crypter un cookie d'authentification Content Server. Si aucune clé n'est définie dans ce champ, Content Server utilise une clé par défaut. Toutefois, OpenText vous recommande vivement de modifier cette valeur et d'utiliser une clé unique.

Clé de codage des données

Content Server utilise également une clé de codage des données, qui vous permet de générer une chaîne de texte à utiliser comme clé pour crypter le mot de passe de l'administrateur de la base de données lorsqu'il est inclus dans une requête Content Server. Si aucune clé n'est définie dans ce champ, Content Server utilise une clé par défaut. Toutefois, OpenText vous recommande vivement de modifier cette valeur et d'utiliser une clé unique.

Informations relatives à l'authentification des cookies

Lors de la configuration du cookie d'authentification, vous pouvez inclure l'adresse IP du client en tant que valeur. La liste Adresse IP client vous permet d'appliquer un masque de bits lors de la comparaison des adresses IP de différentes requêtes. Le masque de bits indique la portion de l'adresse IP à comparer. Les éléments non nuls de l'adresse IP sont comparés à l'adresse IP placée dans le cookie lors de la connexion. Par exemple, si vous sélectionnez 255.255.0.0, seuls les deux premiers caractères sont comparés. Ceci permet de vérifier que les requêtes proviennent du même réseau, sans nécessiter d'adresses IP identiques. Par défaut, ce champ est défini sur 255.255.255.255 (Compare Entire IP Address).

Vous pouvez choisir les attributs d'utilisateur pour les inclure dans le cookie d'authentification. Les options incluent l'ID de propriétaire, qui est l'ID de l'utilisateur qui a créé l'utilisateur, la Date d'expiration du mot de passe et le Mot de passe codé dans un sens.

Vous pouvez également définir un délai d'expiration pour les cookies d'authentification. Après avoir défini l'intervalle, il est nécessaire de se connecter à nouveau. L'expiration des cookies impliquant une interaction supplémentaire avec la base de données, elle peut avoir un impact sur les performances. Par défaut, les cookies sont définis pour expirer 30 minutes après la réalisation de la dernière action.

Date d'expiration des cookies de connexion

Vous pouvez définir la date d'expiration des cookies pour les gérer dans la page de connexion Content Server. Ce paramètre contrôle l'affichage de la langue lorsque des utilisateurs ont installé plusieurs lots de langue. Un cookie sur la page de connexion Content Server retient la sélection de langues d'un utilisateur jusqu'à son authentification. Il utilise ensuite sa langue préférée.

La date d'expiration des cookies est calculée en fonction de la date et de l'heure de l'ordinateur de l'utilisateur et non de la date et de l'heure de Content Server. Le nombre de jours doit être un entier positif compris entre 1 et 999. Par défaut, les cookies de connexion sont définis pour expirer après 8 jours.

Stratégies de connexion

Cette section aide à sécuriser Content Server contre les échecs répétés de tentative de connexion utilisateur, l'ouverture de plusieurs sessions simultanément et par le même utilisateur et la falsification de requêtes intersites.

  • Activer Désactiver les sessions simultanées depuis plusieurs ordinateurs, qui empêche les utilisateurs d'accéder simultanément à Content Server à partir de plusieurs ordinateurs. Si vous activez cette fonction, les utilisateurs peuvent se connecter à Content Server à partir d'un ordinateur, mais s'ils se connectent une seconde fois à partir d'une autre machine, la première session Content Server est interrompue.

    Dans le champ excepté pour les hôtes, vous pouvez fournir une liste d'adresses IP séparées par des virgules pour les ordinateurs auxquels cette fonction ne s'applique pas. De plus, vous pouvez spécifier un caractère générique unique * pour indiquer plusieurs ordinateurs. Par exemple, l'entrée suivante est valide : 123.45.*,192.*,123.99.2.51. Par défaut, cette option est désactivée.

  • Le paramètre Autoriser la connexion par requête HTTP spécifie le comportement de connexion par défaut. Lorsque cette option est activée, les utilisateurs sont autorisés à se connecter à Content Server à l'aide d'une requête HTTP, qui stocke leurs informations d'authentification dans un signet. Lorsque cette option est désactivée, les utilisateurs doivent se connecter à nouveau lorsque les cookies de cette session expirent. Cette option est désactivée par défaut.

  • Lorsqu'il est activé, le paramètre Nécessite un jeton de requête sécurisé permet d'empêcher la falsification de requêtes intersites : une attaque par l'intermédiaire de laquelle un utilisateur lance involontairement une action dans un logiciel auquel se dernier est déjà connecté. Le jeton de requête sécurisé est une valeur partagée entre le serveur et le navigateur lorsque l'utilisateur réalise certaines actions. Cette valeur doit accompagner les requêtes pour d'autres actions. Si la valeur n'est pas présente, l'action demandée n'est pas valide. Cette option est désactivée par défaut.

Nouvelles tentatives de saisie du mot de passe

Cette section définit le nombre de fois où un mot de passe incorrect peut être saisi par un Utilisateur, Administrateur Web et Administrateur Content Server avant que la connexion ne soit désactivée et s'il faut envoyer un e-mail à l'administrateur. Par défaut, ces options sont activées.

Les stratégies de connexion comprennent des options pour alerter par e-mail les administrateurs au sujet d'exceptions de stratégie, comme spécifié sur la page Configurer les paramètres du serveur. Si cette adresse n'est pas spécifiée, le message de notification n'est pas envoyé. Pour plus d'informations sur la spécification de l'adresse e-mail de l'administrateur, reportez-vous à la rubrique Configuration des paramètres de base du serveur.

  • Utilisateur

    Un compte utilisateur peut être désactivé lorsque trop de tentatives de connexion sont effectuées. La fonction de compte désactivé ne s'applique pas à l'accès Directory Services au système car des services de déverrouillage sont disponibles dans la plupart des mises en œuvre Directory Services. Dans ce cas, le compte Content Server n'est pas désactivé, mais le compte Directory Service ou système est verrouillé.

    • Désactiver la connexion en cas de saisie d'un mot de passe incorrect, qui vous permet de spécifier les actions à effectuer lors de l'échec de la connexion d'un utilisateur.

    • Nombre de tentatives de connexion autorisées avant que le compte ne soit désactivé. Par défaut, ce paramètre est activé et le nombre d'échecs de tentatives de connexion est défini sur 5.

    • Envoyer un e-mail à l'administrateur lorsque la connexion est désactivée, qui vous permet de spécifier si l'administrateur Content Server reçoit un e-mail dès que le nombre spécifié de tentatives de connexion est dépassé.

  • Administrateur Web

    Si trop de tentatives de connexion infructueuses sont effectuées pour saisir le mot de passe de l'administrateur Web, la saisie de ce dernier peut être désactivée pendant un certain temps afin de diminuer les risques de découverte de mot de passe via une attaque en force brute.

    • Désactiver la connexion en cas de saisie d'un mot de passe incorrect, qui vous permet de spécifier les actions à effectuer lors de l'échec de la connexion d'un administrateur Web.

    • Nombre de tentatives de connexion autorisées avant que le compte ne soit désactivé. Par défaut, ce paramètre est activé et le nombre d'échecs de tentatives de connexion est défini sur 5.

    • Nombre de minutes de désactivation de la connexion, qui vous permet de spécifier combien de temps l'administrateur Web doit attendre avant de tenter de se connecter à nouveau. Par défaut, ce paramètre est activé et le nombre de minutes est défini sur 5.

    • Envoyer un e-mail à l'administrateur lorsque la connexion est désactivée, qui vous permet de spécifier si l'administrateur Content Server doit recevoir un e-mail dès que le nombre spécifié de tentatives de connexion est dépassé.

  • Admin

    Le compte administrateur est essentiel et ne peut pas être désactivé sans affecter le système. Cette stratégie vous permet d'alerter les administrateurs lorsqu'une personne tenterait de deviner le mot de passe.

    • Envoyer un e-mail à l'administrateur en cas d'échecs de saisie du mot de passe excessifs, qui vous permet de spécifier si l'administrateur Content Server doit recevoir un e-mail dès que le nombre spécifié de tentatives de connexion est dépassé.

    • Maximum d'échecs de saisie du mot de passe, qui vous permet de spécifier le nombre de fois où un administrateur peut tenter de se connecter à Content Server avant l'envoi d'un e-mail. Par défaut, ce paramètre est activé et le nombre d'échecs de tentatives de connexion est défini sur 5.

       

      Note :
      La fonction de désactivation de connexion ne s'applique pas au compte Admin Content Server. Vous pouvez protéger le compte administrateur Content Server en autorisant uniquement l'accès à certaines adresses IP client. Pour plus d'informations sur la limitation des tentatives de connexion du compte Admin Content Server par adresse IP, reportez-vous à la rubrique Limitation des connexions du compte administrateur par adresse IP.

       

Configuration de la page Connexion

Cette section permet de s'assurer de la sécurité de la connexion de l'utilisateur. Par défaut, cette option est désactivée. La section Configuration de la page de connexion comprend les paramètres suivants :

  • La case à cocher Désactiver l'exécution automatique du navigateur, qui vous permet d'ajouter un attribut à la page Identification de l'utilisateur afin de demander aux navigateurs Web clients de ne pas stocker ni exécuter de manière automatique les informations relatives à l'identification de l'utilisateur pour Content Server. Par défaut, cette option est activée, ce qui signifie que les utilisateurs doivent saisir leurs informations d'authentification complètes à chaque fois qu'ils se connectent à Content Server.

     

    Note :
    Certains navigateurs risquent de ne pas interpréter correctement cet attribut.

     

  • Le champ Nombre de jours d'utilisation du mot de passe, qui vous permet d'activer les cookies d'authentification permanents et de définir leur délai d'expiration. Ce paramètre ajoute une case à cocher Retenir le mot de passe sur la page Identification de l'utilisateur. Un cookie d'authentification permanent sera enregistré pendant le nombre de jours défini sur l'ordinateur des utilisateurs qui activent cette case à cocher lors de la connexion.

     

    Note :
    La désactivation de ce paramètre ne supprime pas le cookie d'authentification permanent des navigateurs clients qui en sont déjà pourvus. Les cookies existants sont conservés jusqu'à leur expiration. D'ici là, ils permettent l'accès à Content Server. Pensez-y quand vous devrez décider de la durée des cookies d'authentification.

     

Filtrage des arguments de requête

Lorsque le filtrage des arguments de requête est activé, toutes les requêtes client de Content Server sont comparées aux valeurs du champ Filtrer la chaîne. Par défaut, ces champs sont vides. En cas de correspondance, la requête est rejetée.

La liste de filtres est délimitée par un séparateur à définir. Si le séparateur utilisé est une virgule (,) par exemple, saisissez trois filtres comme suit :

<Filtre_A>,<Filtre_B> ,<Filtre_C>

Le séparateur peut être configuré de manière à prévenir tout conflit avec les filtres souhaités. Par exemple, si vous souhaitez utiliser une barre oblique inverse comme séparateur, saisissez :

<Filtre_A>\<Filtre_B> \<Filtre_C>

Affichage de la taille du recueil

La case à cocher Masquer le nombre d'éléments dans le champ Affichage de la taille du recueil empêche les utilisateurs d'afficher le nombre d'éléments contenus dans un recueil.

 

Note :
Ce paramètre s'applique également aux prospecteurs. Lorsque ce paramètre est défini, les nœuds prospecteurs affichent une valeur vide pour la taille.

 

Expiration du jeton de requête sécurisée

Le champ Expiration du jeton de requête sécurisée fait partie du cadre de sécurité améliorée qui permet aux requêtes d'accéder à la fonctionnalité de génération et de vérification des jetons d'authentification (cookie). Un jeton peut être passé sur des requêtes séquentielles et assurer que les vérifications de sécurité appropriées ont lieu. Le paramètre par défaut est une expiration du délai après 300 secondes.

Hôtes client Content Server

Le champ Hôtes client Content Server contient les adresses IP des serveurs à partir desquels les requêtes doivent être acceptées. Par défaut, ce champ est vide et toutes les connexions client sont acceptées.

Sites Web de référence sécurisés

Le paramètre de Sites Web de référence sécurisés est une mesure de sécurité permettant d'empêcher les autres utilisateurs d'exploiter votre site Web. Les sites dans cette liste apparaissent dans un en-tête de référence HTTP qui valide les requêtes HTTP provenant d'un autre site Web. Si une requête provient d'un site Web non autorisé, l'action n'est pas autorisée et un message d'erreur s'affiche.

 

Note :
Pour Enterprise Process Services Integration, vous devez saisir le paramètre Sites Web de référence sécurisés au format suivant : http://<Serveur PW>, où <Serveur PW> est l'ordinateur où Process Workplace (le serveur Web faisant partie des services de processus Entreprise) est exécuté.

 

Fonctions des documents

La zone Fonctions des documents contient des boutons radio permettant d'activer ou de désactiver les fonctions d'ouverture et d'affichage en tant que page Web dans Content Server. Lorsque ces fonctions sont activées, les utilisateurs peuvent ouvrir un document dans son application native, ou les afficher en tant que pages Web, en cliquant sur le lien du document. Lorsqu'elle est désactivée, l'option Ouvrir n'apparaît pas dans le menu Fonctions d'un document ou sur la page Vue d'ensemble. La fonction Ouvrir document est désactivée par défaut.

Interdomaines sécurisés

Le champ Inter-domaines sécurisés vous permet de gérer Content Server et des intégrations d'applications Web tierces. Vous enregistrez des pairs <clé> ; <cible> où <clé> est une valeur devant être transmise sous forme de paramètre d'URL et qui exécutera la recherche d'enregistrement et <cible> est un chemin d'URL enregistré pointant vers une ressource cible (application Web tierce).

 

Note :
Pour Enterprise Process Services Integration, vous devez saisir le paramètre Inter-domaines sécurisés au format suivant : pw;http://< serveur PW>/pw/client/csbrowse.htm, où <Serveur PW> est l'ordinateur où Process Workplace (le serveur Web faisant partie des services de processus Entreprise) est exécuté.
