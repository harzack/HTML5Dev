Chaque site Content Server possède un serveur et un serveur d'administration. Le serveur contrôle toutes les fonctionnalités et affiche les pages que vous voyez dans l'interface utilisateur à l'aide du serveur HTTP. Le serveur d'administration gère les processus de flux de données dans le système (par exemple les processus Extracteur Entreprise et Directory Walker et les processus Conversion de documents et Distributeur de mise à jour). Il gère également un système de mise en mémoire cache de fichier qui permet d'améliorer l'efficacité de certaines opérations Content Server, telles que la mise en évidence des résultats.

 

Note :
Sur chaque site Content Server, le serveur et le serveur d'administration sont représentés comme des services dans votre système d'exploitation Windows.

 

Après avoir installé Content Server, vous devez configurer les serveurs d'administration sur votre site. Vous pouvez également configurer la mémoire cache du fichier du serveur d'administration. Vous pouvez ensuite effectuer des opérations de maintenance, telles que la consultation, l'exécution d'une commande ping, l'interruption ou la réinitialisation du serveur d'administration. Si nécessaire, vous pouvez également resynchroniser un serveur d'administration pour vous assurer que ses données reflètent avec exactitude les informations stockées dans la base de données Content Server.

La resynchronisation d'un serveur d'administration modifie le fichier otadmin.cfg afin qu'il prenne en compte les informations actuellement stockées dans la base de données Content Server et ordonne au serveur d'administration de créer ou de supprimer des objets, le cas échéant, en vue de prendre en compte les informations mises à jour.

Vous pouvez resynchroniser un serveur d'administration à tout moment. Toutefois, Content Server vous invite automatiquement à resynchroniser un serveur d'administration après avoir modifié son nom d'hôte et son numéro de port.

Avant de resynchroniser un serveur d'administration, assurez-vous que les chemins de répertoire et les numéros de port enregistrés pour les objets source de données de la base de données Content Server sont corrects sur l'hôte du serveur d'administration. Pour ce faire, allez sur l'onglet Spécifique de la page Propriétés de chaque objet source de la base de données et vérifiez que les chemins et les numéros de port sont corrects sur l'hôte du serveur d'administration.

Le serveur stocke les informations concernant ses serveurs d'administration dans la base de données Content Server, y compris les processus de flux de données et les moteurs de recherche qu'il contrôle en utilisant ces serveurs. Les serveurs d'administration n'étant pas directement connectés à la base de données Content Server, chacun d'entre eux dispose de sa propre copie de ces informations dans le fichier otadmin.cfg.

 

Note :
Le fichier otadmin.cfg est stocké dans le répertoire config d'une installation Content Server principale.

 

Les informations du fichier otadmin.cfg d'un serveur d'administration doivent toujours correspondre aux informations de la base de données Content Server. Cependant, il peut arriver que ces informations ne correspondent plus. Par exemple :

  • Vous changez le nom d'hôte ou le numéro de port d'un serveur d'administration.

  • Vous effectuez une installation Content Server sur un nouvel hôte et vous la connectez à une base de données Content Server créée sur un autre hôte. Les informations relatives aux emplacements des objets source de données qui sont stockées dans la base de données Content Server ne sont peut-être pas correctes pour le nouvel hôte si les répertoires sont mappés ou montés différemment, en cas de conflit de numéros de port ou si les serveurs d'administration enregistrés ne sont plus disponibles.

  • Le fichier otadmin.cfg d'un serveur d'administration enregistré est corrompu ou a été supprimé.

  • Les modifications apportées aux informations de la base de données Content Server n'ont pas pu être appliquées au fichier otadmin.cfg d'un serveur d'administration. Ce cas rare peut se présenter si vous modifiez ces informations en créant des commandes SQL directement dans la base de données Content Server.

  • Le serveur d'administration est passé en mode sans échec. Pour en savoir plus, consultez le document Maintenance des serveurs d'administration.

Si vous pensez que les informations relatives aux serveurs d'administration et aux sources de données stockées dans la base de données Content Server ne correspondent pas aux informations stockées dans les fichiers otadmin.cfg des serveurs d'administration enregistrés, vous devez les resynchroniser. La resynchronisation d'un serveur d'administration vérifie la stabilité des index du serveur afin de s'assurer qu'ils ne sont pas corrompus, modifie le fichier otadmin.cfg afin qu'il prenne en compte les informations actuellement stockées dans la base de données Content Server, puis ordonne au serveur d'administration de créer ou de supprimer des objets, le cas échéant, en vue de prendre en compte les informations mises à jour. Cependant, si un ou plusieurs index sont corrompus, la resynchronisation s'arrête et affiche un message d'erreur vous demandant de contacter le service d'assistance clientèle d'OpenText.

La resynchronisation d'un serveur d'administration donne parfois des résultats inattendus. Par exemple, s'il existe une différence entre la base de données Content Server et le fichier otadmin.cfg d'un serveur d'administration concernant la présence ou non d'un index spécifique sur l'hôte du serveur d'administration, il y a deux résultats possibles, selon les circonstances :

  • S'il existe une entrée pour l'index dans le fichier otadmin.cfg d'un serveur d'administration (ce qui signifie que l'index réside sur l'hôte de ce serveur), mais qu'il n'y a pas d'entrée correspondante dans la base de données Content Server, la resynchronisation ordonne au serveur d'administration de supprimer cette entrée du fichier otadmin.cfg et de supprimer l'index de l'emplacement erroné.

  • Si les informations de la base de données Content Server indiquent que l'index existe sur un hôte spécifique du serveur d'administration mais que le fichier otadmin.cfg de cet hôte ne contient pas d'entrée pour cet index, la resynchronisation ordonne au serveur d'administration d'ajouter une entrée pour l'index dans le fichier otadmin.cfg et de créer un index vide à l'emplacement spécifié dans la base de données Content Server.

Processus inconnus ou orphelins

Une fois la resynchronisation terminée, un résumé des résultats s'affiche. Si certains processus dans le fichier otadmin.cfg ne correspondent pas aux informations stockées dans la base de données, ils sont répertoriés dans la section des résultats Processus inconnus ou orphelins.

Des erreurs peuvent se produire dans cette section étant donné la présence d'une restriction connue dans le processus de synchronisation du serveur d'administration qui empêche parfois certains processus de se resynchroniser correctement. Une double synchronisation résout ce problème potentiel.

Si des erreurs surviennent, exécutez le processus de resynchronisation une deuxième fois pour les réparer. Aucune erreur ne devrait se produire après une seconde resynchronisation, par conséquent, si certaines erreurs sont toujours présentes, contactez le service d'assistance clientèle d'OpenText.
