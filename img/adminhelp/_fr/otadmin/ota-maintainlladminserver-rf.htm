Après avoir configuré vos serveurs d'administration, vous pouvez effectuer des opérations de maintenance. Vous pouvez par exemple parcourir un serveur d'administration pour déterminer les processus de flux de données qu'il gère, vous pouvez exécuter une commande ping vers un serveur d'administration pour déterminer s'il est connecté et vous pouvez interrompre un serveur d'administration si vous voulez qu'il soit exécuté en mode sans échec pendant que vous effectuez d'autres opérations d'administration. Si vous interrompez un serveur d'administration, vous devez le réinitialiser pour qu'il ne soit plus en mode sans échec. Vous pouvez également afficher les numéros de port associés aux processus d'indexation et de recherche gérés par un serveur d'administration.

Parcourir les serveurs d'administration

Les sites Content Server peuvent avoir plusieurs serveurs d'administration différents, chacun d'entre eux pouvant gérer différents composants d'index (par exemple, les processus de flux de données et les moteurs de recherche) dans la base de données Content Server. Vous pouvez parcourir un serveur d'administration pour consulter la liste des composants d'index qu'il gère.

Si un site Content Server dispose de plusieurs serveurs d'administration, vous pouvez les parcourir afin de déterminer comment les processus de flux de données et les moteurs de recherche sont distribués entre les serveurs. Vous pouvez également surveiller le statut et gérer les processus ou les moteurs de recherche. Les tâches d'administration que vous pouvez effectuer sont les mêmes que celles que vous pouvez effectuer sur les processus ou les moteurs de recherche lorsque vous accédez à leur emplacement dans le système Content Server.

Exécution d'une commande ping vers des serveurs d'administration

Lorsqu'un serveur d'administration n'est plus disponible (c'est-à-dire que, pour une raison quelconque, l'administrateur Content Server le déconnecte), le serveur exécute une commande ping trois fois afin de déterminer s'il est de nouveau disponible. Si la commande échoue trois fois, le serveur considère le serveur d'administration comme étant définitivement déconnecté et n'exécute pas d'autres commandes ping. Lorsque cela se produit et que le serveur d'administration est de nouveau disponible, vous devez exécuter vous-même une commande ping afin de reconnecter ce serveur au serveur.

Interruption des serveurs d'administration

Vous devez interrompre un serveur d'administration pour pouvoir l'exécuter en mode sans échec afin que vous puissiez effectuer des tâches de maintenance du système sans gêner ses fonctions principales.

Mode sans échec

Le serveur d'administration adopte un mode sans échec de dépannage car :

  • L'administrateur émet la commande go safe.

  • Content Server ne peut pas enregistrer un fichier de configuration ou un fichier de processus actif.

Un serveur d'administration passe en mode sans échec lorsqu'il ne peut pas écrire dans son fichier otadmin.cfg ou dans son fichier otadmin.pid. Lorsqu'un serveur d'administration passe en mode sans échec, son statut s'affiche sur la page Volume des objets système. L'onglet Général de la page Propriétés du serveur d'administration affiche un message plus détaillé décrivant la condition qui a provoqué le passage en mode sans échec du serveur d'administration.

Ces commandes sont conformes en mode sans échec :

  • exit process

  • quit

  • reset server

  • shutdown

  • status process

  • status ipool

  • status server

  • statuscode process

  • statustext process

  • sync

Un serveur d'administration ne vérifie pas la quantité d'espace disque disponible et ne passe donc pas en mode sans échec lorsque l'espace disque est faible.

Réinitialisation des serveurs d'administration

Vous devez réinitialiser un serveur d'administration pour qu'il ne soit plus en mode sans échec et pour qu'il soit de nouveau actif.

Affichage des numéros de port

Affichez les numéros de port pour déterminer quels ports sont en cours d'utilisation ou pour vous assurer que tous les processus d'indexation et de recherche gérés par un serveur d'administration sont exécutés sur des ports uniques. La liste des numéros de port comprend les numéros de port utilisés par les processus suivants (s'ils existent) :

  • Extracteur Content Server

  • Director Walker

  • XML Activator

  • Conversion de documents

  • Moteur d'indexation

  • Moteur de recherche

  • Fédérateur de recherche

  • Distributeur de mise à jour

  • Serveur d'administration

  • Serveur DCS

Vous pouvez trier cette liste dans l'ordre ascendant ou descendant des numéros de port, par type d'objet, par nom ou par emplacement. Les numéros de port affichés en rouge indiquent un conflit de ports (c'est-à-dire qu'au moins deux processus sont configurés pour être exécutés sur le même port). Dans ce cas, vous pouvez effectuer le tri par numéro de port pour identifier l'origine du conflit, puis modifier les valeurs des ports en conflit.

 

Note :
Cette liste comprend uniquement les ports utilisés par les processus d'indexation et de recherche et non l'ensemble des ports Content Server. Pour savoir quels autres ports sont utilisés par Content Server, vous pouvez consulter le fichier opentext.ini du site. Si vous exécutez Content Server dans un environnement de clusters, vous devez consulter tous les fichiers opentext.ini du cluster.

 

Mise à jour des chemins des processus de recherche

La page Gestion de chemin de serveur d'administration vous permet de mettre à jour les mappages et les noms de chemin pour la configuration de la recherche afin de répercuter toute modification apportée au cours du processus de mise à niveau. Ces chemins doivent être mis à jour une fois pour chaque serveur d'administration dans le système.
