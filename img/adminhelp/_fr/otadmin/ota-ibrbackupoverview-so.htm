L'index d'une source de données permet aux utilisateurs Content Server de rechercher des éléments dans cette source de données. Si un index a été supprimé ou si vous pensez que son contenu est corrompu, vous pouvez le recréer en éliminant son flux de données et en réindexant sa source de données. Cependant, dans le cas d'une source de données importante, telle que l'espace de travail Entreprise de Content Server, les opérations d'élimination et de réindexation peuvent prendre du temps. Nous vous conseillons donc de sauvegarder régulièrement vos index. Les sauvegardes d'index vous permettent de restaurer un index tel qu'il était auparavant sans avoir à effectuer une réindexation complète. Les opérations de sauvegarde et de restauration agissant uniquement sur l'index, elles sont beaucoup plus rapides que la réindexation d'une source de données importante.

Content Server sauvegarde un index en copiant tous les fichiers nécessaires de l'emplacement de l'index dans un répertoire de destination. Vous pouvez ensuite utiliser le logiciel de sauvegarde de votre société pour effectuer une copie de sauvegarde du répertoire de destination. Lorsque vous restaurez l'index, vous copiez les fichiers appropriés dans un répertoire et indiquez cet emplacement au processus de restauration.

Dans Content Server, il existe deux types de sauvegarde que vous pouvez utiliser en même temps :

  • Complet : Une sauvegarde de type « Complet » permet d'enregistrer l'intégralité de l'index dans un répertoire de destination. Toute sauvegarde existant précédemment dans cet emplacement est écrasée. Vous ne pouvez conserver qu'une seule sauvegarde de ce type au même emplacement pour un même index.

  • Différentiel : Une sauvegarde de type « Différentiel » permet d'enregistrer uniquement les données qui ont changé dans l'index depuis la sauvegarde précédente. Dans la plupart des cas, cela signifie que ce type de sauvegarde nécessite moins d'espace disque que la sauvegarde de type « Complet » ; cependant, Content Server fusionnant à intervalle régulier, cela n'est pas toujours le cas. Supposons par exemple qu'il existe quatre fragments d'index lorsque vous effectuez votre première sauvegarde complète. La sauvegarde complète enregistre les quatre fragments d'index. Supposons ensuite qu'une opération de fusion a lieu. Vous avez alors un fragment d'index qui contient les quatre fragments d'index précédents. Si vous effectuez une sauvegarde différentielle à ce moment-là, le fragment d'index restant est enregistré, mais il nécessitera peut-être autant, voire plus, d'espace disque que la sauvegarde complète précédente.

Mise en place d'une stratégie de sauvegarde

Pour profiter pleinement des fonctions de sauvegarde et de restauration, vous devez élaborer une stratégie qui déterminera la fréquence et les types de sauvegarde qui sont appropriés pour votre site. Pour la plupart des sites, OpenText recommande de planifier des sauvegardes complètes moins régulièrement et des sauvegardes différentielles plus régulièrement. Vous pouvez par exemple planifier une sauvegarde complète une fois par semaine et des sauvegardes différentielles tous les soirs.

Automatisation des sauvegardes d'index

Un gestionnaire de sauvegarde vous permet de programmer des processus de sauvegarde automatisés afin de mettre en place une stratégie de sauvegarde. Un gestionnaire de sauvegarde est un dossier Content Server spécial contenant des processus de sauvegarde planifiés qui sont associés à une source de données spécifique. Dans le gestionnaire de sauvegarde, vous pouvez créer, modifier ou supprimer des processus de sauvegarde programmés. Avant de pouvoir sauvegarder l'index d'une source de données, cette dernière doit disposer d'un gestionnaire de sauvegarde. Si la source de données n'en a pas, vous devez en créer un.

Sauvegarde manuelle des index

Vous pouvez à tout moment démarrer des processus de sauvegarde de façon manuelle à l'aide de l'Assistant de sauvegarde, à condition qu'aucun autre processus de sauvegarde ne soit en cours d'exécution sur la source de données que vous voulez sauvegarder. Si vous essayez de démarrer de façon manuelle un processus de sauvegarde sur une source de données qui ne dispose pas de gestionnaire de sauvegarde, Content Server vous invite à en créer un.

 

Note :
OpenText vous recommande d'automatiser les sauvegardes d'index.

 

Affichage de l'historique des sauvegardes

Si vous voulez voir les sauvegardes précédemment effectuées sur une source de données, vous pouvez consulter l'historique des sauvegardes.

Restauration des index

Si un index est endommagé ou corrompu et que vous l'avez sauvegardé, vous pouvez utiliser l'Assistant de restauration pour rétablir l'index tel qu'il était au moment de la sauvegarde la plus récente. Vous pouvez ensuite démarrer le processus d'extraction ou producteur pour cette source de données afin d'extraire toutes les modifications de la base de données depuis la dernière sauvegarde.

Avant de pouvoir restaurer un index, vous devez récupérer les sauvegardes de cet index depuis le référentiel de sauvegarde de votre société et copier ces fichiers dans un emplacement temporaire. L'Assistant de restauration de Content Server replace ensuite ces fichiers dans l'emplacement principal de l'index. Vous devez également arrêter les processus d'indexation et de recherche ainsi que le gestionnaire de sauvegarde avant de restaurer un index. Cela permet d'éviter qu'une sauvegarde automatique (programmée) se produise lors de l'opération de restauration.
