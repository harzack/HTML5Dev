OpenText Object Importer permet d'importer automatiquement des fichiers du système de fichiers local vers Content Server. Il est possible de configurer le fichier de contrôle qui gère le processus. Le fichier de contrôle permet de spécifier vers quel emplacement de Content Server sont téléchargés les objets et l'endroit du système de fichiers où se trouvent les objets à télécharger. Vous disposez également du contrôle total des autorisations, titre, attributs et autres métadonnées de chaque objet dans Content Server. Utilisation de la configuration de l'importateur d'objet

La page Configurer l'importateur d'objet permet de définir comment se déroulent les processus d'importation d'objets. Vous devez spécifier les chemins complets vers les répertoires de fichier de contrôle, de travail et de journaux sur la page Configurer l'importateur d'objet. Vous devez d'abord créer les trois répertoires et vous assurer qu'ils sont visibles pour Content Server :

  • L'importateur d'objet recherche les fichiers d'importation dans le répertoire du fichier de contrôle. Voici un exemple de répertoire de fichier de contrôle valide : C:\OPENTEXT\object_importer\controlfile, où C:\OPENTEXT est la valeur par défaut pour Content Server.

  • Le répertoire de travail est l'emplacement où l'importateur d'objet traite l'importation. Voici un exemple de répertoire de fichier de travail valide : C:\OPENTEXT\object_importer\working, où C:\OPENTEXT est la valeur par défaut pour Content Server.

  • Le répertoire des journaux est l'emplacement où l'importateur d'objet copie les fichiers journaux. Pour afficher et supprimer des fichiers journaux de la page Importateur d'objet Afficher les fichiers journaux d'importation, ce répertoire doit se trouver dans <Content Server_home>\logs ou <Content Server_home>\object_importer. Voici un exemple de répertoire de fichier journal valide : C:\OPENTEXT\object_importer\logfiles, où C:\OPENTEXT est la valeur par défaut pour Content Server.

 

Important :
Les répertoires de fichier de contrôle, de travail et de journaux doivent être créés et visibles pour Content Server. Vous devez les spécifier sur la page Configurer l'importateur d'objet pour que les importations puissent avoir lieu.

 

En plus de la configuration spécifique à l'importateur d'objet détaillée plus haut, vous devez également :

  • Vous assurer que l'adresse e-mail de l'utilisateur Admin est définie dans la page du compte de l'utilisateur Admin. Vous accédez à cette page en sélectionnant Utilisateurs et groupes dans le menu Entreprise. Recherchez l'utilisateur Admin sur la page Utilisateurs et groupes, puis cliquez sur Modifier en regard du nom de l'utilisateur Admin. Entrez l'adresse e-mail de l'utilisateur Admin.

  • Complétez la page Notifications. Il n'est pas nécessaire que la page Notifications soit activée. Pour plus d'informations sur la page Notifications, reportez-vous à la rubrique Configuration des notifications, des rapports, des événements et des paramètres dans Suppression des événements en suspens.

 

Important :
Lorsque l'adresse e-mail de l'utilisateur Admin n'est pas définie ou lorsque la page Notifications n'est pas complétée, les importations n'ont pas lieu.

 

Utilisation de l'administration de la base de données Importateur d'objet

Utilisez la page Administration des données pour gérer le schéma de la base de données requis pour cette application.

Utilisation de l'état de l'importation

La page Etat de l'importation vous permet de vérifier l'état des threads d'importation en cours d'exécution. Si aucun thread d'importation n'est en cours d'exécution lorsque vous cliquez sur le lien Etat de l'importation, Content Server affichera une liste d'état vide.

Utilisation de l'importation manuelle d'objet de l'importateur d'objet

La page Importation manuelle d'objet vous permet d'importer manuellement des documents vers Content Server plutôt que de procéder à une importation planifiée. L'importation manuelle est généralement utilisée dans les environnements de test. Les fichiers de contrôle utilisés lors d'une importation manuelle ne sont jamais supprimés.

Lorsque vous procédez à une importation manuelle d'objet, il n'est pas nécessaire que le fichier de contrôle se trouve sur la machine où Content Server est installé.

 

Note :
Lors des importations manuelles d'un grand nombre de fichiers, il est possible que lors de l'ajout de nombreux objets à Content Server, le navigateur signale une erreur ou semble planter. Cela ne veut toutefois pas dire que l'importation a échouée.

  • Si vous voyez s'afficher une boîte de dialogue d'erreur, cliquez sur OK puis sur le lien Accueil Admin. Sur la page Administration de l'importateur d'objet, cliquez sur le lien Afficher l'état de l'importation pour visualiser l'état de l'importation.

  • Si le navigateur semble ne pas fonctionner et qu'aucun message d'erreur n'est généré, sur la page Administration de l'importateur d'objet, cliquez sur le lien Afficher l'état de l'importation pour visualiser l'état de l'importation.

 

Utilisation de la programmation de l'importateur d'objet

La page Programmer les tâches d'importation vous permet définir quels agents de programmation sont activés ainsi que leur programmes respectifs. Les notifications Content Server doivent être configurées avant de programmer des importations automatiques. Pour plus d'informations, consultez la section Utilisation de la configuration de l'importateur d'objet.

Le module Object Importer s'exécute sur un seul thread. De ce fait, une seule instance de l'importateur peut s'exécuter sur chaque installation Content Server. Le module Object Importer peut être installé sur plusieurs machines connectées à la même base de données Content Server afin de réduire le temps requis pour importer de grandes quantités de données. Les informations de la page Programmer les tâches d'importation s'appliquent à toutes les instances Content Server connectées à la même base de données. Chaque instance de Content Server, jusqu'à un maximum de dix, peut exécuter son propre agent d'importation unique.

Si vous exécutez plusieurs instances de Content Server, vous devrez ajouter ce qui suit à la section [scheduleactivity] dans le fichier opentext.ini de chaque serveur :

2990=0
2991=0
2992=0
2993=0
2994=0
2995=0
2996=0
2997=0
2998=0
2999=0

Redémarrez chaque serveur après avoir modifié et enregistré le fichier opentext.ini . Notez le nom de la machine et l'ID d'agent pour vous y référer ultérieurement.

Activez un agent différent par machine en définissant l'ID d'agent égal à un, par exemple 2991=1. En général, lorsque plusieurs instances de Content Server sont déployées, les agents de programmation sont désactivés sur toutes les machines sauf une. Le module Object Importer requiert que les agents de programmation soient activés sur toutes les machines devant importer des objets. D'autres ID d'agent peuvent être désactivés en définissant leur valeur sur zéro.

Ce qui suit doit être ajouté au fichier opentext.ini sur toutes les machines sauf celle qui fait office d'agent :

[options]
EnableAgents=TRUE

[loader]
load=sockserv;agents

[scheduleactivity]
1000=0
4000=0
8900=0
8999=0
9000=0
3502=0

Redémarrez chaque serveur après avoir modifié et enregistré le fichier opentext.ini . Notez le nom de la machine et l'ID d'agent pour vous y référer ultérieurement.

Pour plus d'informations sur la modification du fichier opentext.ini, consultez la section Compréhension du fichier opentext.ini dans Compréhension du fichier opentext.ini.

Utilisation de la validation Importateur d'objet

La page Validation vous permet d'exécuter un fichier de contrôle de l'importation à l'aide d'un outil de validation pour rechercher la présence d'erreurs. Cet outil de validation recherche notamment des erreurs d'importation associées à l'importation de métadonnées multilingues.

L'importateur d'objet traite le fichier de contrôle de l'importation pour valider que des conditions spécifiques sont respectées. L'importation effective des données n'aura pas lieu.

Une fois que vous avez sélectionné le fichier de contrôle de l'importation et que vous avez exécuté l'outil de validation sur ce dernier, Content Server affiche une nouvelle page indiquant les résultats de la validation. Cette nouvelle page inclut un lien vers la page Importation manuelle d'objet.

 

Note :
L'outil de validation n'identifie pas toutes les erreurs graves que votre fichier de contrôle de l'importation peut rencontrer. Même après avoir exécuté cet outil et avoir supprimé toutes les erreurs générées par ce dernier, il est possible que l'importation échoue.

 

Utilisation des fichiers journaux Object Importer

La page Afficher les fichiers journaux d'importation répertorie les fichiers situés dans le répertoire des journaux pour une instance particulière de Content Server.

La page Afficher les fichiers journaux d'importation répertorie le nom du fichier temporaire sur le serveur et non celui du fichier de contrôle. Chaque importation crée jusqu'à trois fichiers associés :

  1. Tous les fichiers ayant l'extension .log sont des fichiers journaux. Ils signalent toute erreur intervenue pendant le processus d'importation.

  2. Tous les fichiers qui se terminent par _uncreated.xml contiennent des données du fichier de contrôle qui n'a pas pu être créé.

  3. Tous les fichiers qui se terminent par _unprocessed.xml contiennent des données du fichier de contrôle qui n'a jamais été traité en raison d'une interruption de l'importation.

Les fichiers journaux générés par l'importateur d'objet sont ouverts en mode annexe pour éviter tout remplacement accidentel.

Aucun des fichiers journaux utilisés par l'importateur d'objet ne sera visible sur la page Importateur d'objet Afficher les fichiers journaux d'importation.

Il se peut que vous souhaitiez renommer les fichiers XML non traités ou non créés avant de les réutiliser pour éviter tout problème de convention de dénomination tel que : test_unprocessed_unprocessed.xml
