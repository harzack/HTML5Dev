Balise acl

Description

Utilisez la balise acl pour ajouter, mettre à jour ou supprimer des autorisations pour un objet donné. Si une liste de contrôle d'accès (ACL) existe pour l'objet, les autorisations de cette dernière seront mises à jour. Dans le cas contraire, la liste de contrôle d'accès sera ajoutée à l'objet.

Syntaxe

<acl user="jdoe" permissions="111111100"></acl>

Utilisation

  1. La balise acl n'est pas obligatoire.

  2. La balise acl est facultative dans les cas suivants :

    • <node type="<any_type>" action="create">

    • <node type="<any_type>" action="update">

    • <node type="<any_type>" action="sync">

  3. Si la balise acl est utilisée, deux attributs sont requis :

    1. Soit l'attribut Attribut permissions soit action="remove" .

    2. Soit : Attribut basegroup, Attribut baseowner, Attribut group, Attribut role, Attribut standard soit Attribut user.

Exemple

Pour supprimer l'utilisateur « jdoe » de la liste ACL du dossier « Nouveau dossier » : 

<import>
  <node type="folder" action="update">
    <location>Enterprise:New Folder</location>
    <acl user="jdoe" action="remove"/>
  </node>
</import>

Pour consulter d'autres exemples de balises acl, consultez la section Exemples de fichier de contrôle d'importation de listes de contrôles d'accès.

Attribut basegroup

L'attribut basegroup définit le nom du groupe de base auquel s'applique l'autorisation. Autrement dit, vous pouvez modifier le nom du groupe propriétaire de l'objet et définir les autorisations du groupe propriétaire en une seule étape. La syntaxe de cet attribut est la suivante : 

<acl basegroup="DefaultGroup" permissions="111111111"></acl>

Utilisation
1.

L'attribut basegroup n'est pas obligatoire.

2.

L'attribut basegroup peut être défini sur n'importe quel nom de groupe existant dans Content Server. Les noms de groupe sont sensibles à la casse.

Attribut baseowner

L'attribut baseowner définit le nom du propriétaire de base auquel s'applique l'autorisation. Autrement dit, vous pouvez modifier le nom du propriétaire de l'objet et définir les autorisations du propriétaire en une seule étape. La syntaxe de cet attribut est la suivante : 

<acl baseowner="Admin" permissions="111111111"></acl>

Utilisation
1.

L'attribut baseowner n'est pas obligatoire.

2.

L'attribut baseowner peut être défini sur n'importe quel nom d'utilisateur existant dans Content Server. Les noms d'utilisateur sont sensibles à la casse.

Attribut group

L'attribut group définit le nom du groupe auquel s'applique l'autorisation. La syntaxe de cet attribut est la suivante : 

<acl group="Testing" permissions="111111100"></acl>

Utilisation
1.

L'attribut group n'est pas obligatoire.

2.

L'attribut group peut être défini sur n'importe quel nom de groupe existant dans Content Server. Les noms de groupe sont sensibles à la casse.

Attribut permissions

Description

L'attribut permissions définit les autorisations possibles pour la liste ACL.

Utilisation pour tous les objets sauf pour une discussion

  1. L'attribut permissions est requis si l'attribut action="remove" n'est pas présent.

  2. Pour chaque attribut permissions, vous devez spécifier une chaîne d'autorisations contenant neuf chiffres. Chaque chiffre peut être égal à zéro, ce qui correspond à un refus d'autorisation, ou égal à un, ce qui revient à accorder l'autorisation. Les positions, de un à neuf dans la chaîne, définissent le type d'autorisation pour tous les objets à l'exception des discussions.

    Position Description
    1 Voir le nœud. Définir cette position sur un permet à l'utilisateur ou au groupe spécifié de voir que l'objet existe.
    2 Voir le contenu du nœud. Définir cette position sur un permet à l'utilisateur ou au groupe spécifié de voir que l'objet existe ainsi que son contenu.
    3 Modifier le nœud. Définir cette position sur un permet à l'utilisateur ou au groupe spécifié de voir que l'objet existe ainsi que de le modifier et voir son contenu.
    4 Modifier les autorisations de nœud. Définir cette position sur un permet à l'utilisateur ou au groupe spécifié de voir que l'objet existe ainsi que de le modifier et de voir et modifier son contenu.
    5 Modifier les attributs de nœud. Définir cette position sur un permet à l'utilisateur ou au groupe spécifié de voir que l'objet existe, de le modifier et de voir et modifier son contenu ainsi que de modifier toute métadonnée associée à l'objet.
    6 Ajouter des éléments à un nœud, par exemple ajouter des documents à un dossier. Le sixième chiffre de la chaîne s'applique uniquement aux nœuds de recueil, par exemple un dossier. Lors de la spécification de la balise acl pour un objet non-recueil, tel qu'un document, le sixième chiffre ne s'applique pas. À titre de meilleure pratique, il est conseillé que les objets non-recueil soient définis sur zéro, bien qu'il n'y ait pas de différence en termes de comportement de l'objet après l'importation, quelle que soit la valeur.
    7 Supprimer les versions de nœud. Définir cette position sur un permet à l'utilisateur ou au groupe spécifié de voir que l'objet existe ainsi que de supprimer les versions de cet objet.
    8 Supprimer le nœud. Définir cette position sur un permet à l'utilisateur ou au groupe spécifié de voir que l'objet existe ainsi que de le supprimer.
    9 Réserver le nœud. Définir cette position sur un permet à l'utilisateur ou au groupe spécifié de voir que l'objet existe ainsi que de le réserver.

Utilisation pour une discussion

  1. L'attribut permissions est requis si l'attribut action="remove" n'est pas présent.

  2. L'attribut permissions d'une balise acl appliqué à un objet discussion ne fonctionne pas de la même manière que ceux appliqués aux autres objets.

    Seuls les paramètres d'autorisations répertoriés ci-dessous sont pris en charge pour les discussions :

    Paramètre permissions Description
    permissions="000000000"
    Définit la liste ACL de la discussion sur « Aucun ».
    permissions="111011111"
    Définit la liste ACL de la discussion sur « Ecriture ».
    permissions="110000000"
    Définit la liste ACL de la discussion sur « Lecture ».
    permissions="111111111"
    Définit la liste ACL de la discussion sur « Gérer ».

Attribut role

L'attribut role définit les autorisations pour des groupes spécifiques attribués aux projets dans Content Server. La syntaxe de cet attribut est la suivante : 

<acl role="members" permissions="111111111"></acl>

Utilisation
1.

L'attribut role n'est pas obligatoire.

2.

L'attribut role est disponible de manière facultative lorsque <node type="project" action="<any_action>"> uniquement. L'attribut role peut uniquement être appliqué aux objets enfants des projets et non au projet lui-même. Un dossier est un exemple d'objet enfant d'un projet.

3.

L'attribut role doit se voir attribuer l'une des valeurs suivantes :

  • coordinators : Les autorisations seront attribuées au groupe Coordinateur désigné pour un projet.

  • guests : Les autorisations seront attribuées au groupe Invité désigné pour un projet.

  • members : Les autorisations seront attribuées au groupe Membre désigné pour un projet.

Attribut standard

L'attribut standard permet de définir des autorisations pour le groupe de base (basegroup), le propriétaire de base (baseowner) ou global (world), sans que vous deviez indiquer un utilisateur ou un groupe spécifique. La syntaxe de cet attribut est la suivante : 

<acl standard="owner" permissions="111111100"></acl>

Utilisation
1.

L'attribut standard n'est pas obligatoire.

2.

L'attribut standard doit se voir attribuer l'une des valeurs suivantes :

  • basegroup : Le groupe par défaut d'un nœud.

  • owner : Le propriétaire d'un nœud.

  • world : Tous les utilisateurs et groupes.

Attribut user

L'attribut user définit le nom de l'utilisateur auquel s'applique l'autorisation. La syntaxe de cet attribut est la suivante : 

<acl user="jdoe" permissions="111111110"></acl>

Utilisation
1.

L'attribut user n'est pas obligatoire.

2.

L'attribut user peut être défini sur n'importe quel nom d'utilisateur existant dans Content Server. Les noms d'utilisateur sont sensibles à la casse.
