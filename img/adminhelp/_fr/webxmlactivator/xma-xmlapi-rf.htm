Les fichiers XML générés par vos applications tierces doivent suivre certaines règles pour pouvoir fonctionner correctement avec des processus Création d'un flux de données producteur XML Activator.

Emplacement

Les fichiers XML copiés par une application tierce dans des répertoires pour être lus par un processus XML Activator doivent être entièrement fermés. L'application tierce peut répondre à cette exigence en écrivant les fichiers dans un répertoire local, puis en déplaçant ces fichiers dans le répertoire entrant du processus XML Activator.

Les processus XML Activator répondent également à cette exigence en envoyant des fichiers entièrement fermés vers les répertoires requis par les applications tierces.

Nom

Les processus XML Activator lisent les fichiers dans l'ordre lexicographique (dictionnaire). Cela signifie que les applications tierces doivent nommer leurs fichiers en conséquence pour que le traitement se déroule correctement.

Suppression

Lorsqu'une application tierce a fini de lire les fichiers XML qui lui ont été envoyés par un processus XML Activator, elle doit supprimer ces fichiers de son répertoire de lecture.

Format

En plus de leur contenu (des données binaires ou du texte, par exemple), les fichiers XML générés par des applications tierces doivent inclure des données XML qui correspondent aux messages IPool. Ces données XML indiquent au processus XML Activator ce qu'il doit faire du contenu correspondant. Le tableau suivant décrit les paires Clé IPool-Valeur, que vous incluez dans les fichiers XML comme éléments balisés et qui constituent les messages IPool. Cependant, vous pouvez fournir les données OTURN et Operation sous différents noms de balise si vous spécifiez dans Content Server d'autres mises en correspondance des balises dans les champs Balise d'identificateur et Balise d'opération du processus XML Activator. Vous pouvez configurer ces champs dans l'onglet Infos spécifiques de la page Propriétés de chaque processus XML Activator.

Tableau 1-5 : Paires Clé IPool-Valeur

Clé Balise XML Valeur
OTURN <OTUrn>, à moins que vous n'ayez spécifié une autre balise pour OTURN dans Content Server. Chaîne continuelle et unique qui identifie l'objet de données (contenu inclus dans un fichier XML spécifique). Si vous n'incluez pas cet élément dans un fichier XML, le processus XML Activator en génère un automatiquement. Cependant, l'élément OTURN généré automatiquement n'est pas récupérable. Vous ne pouvez donc pas utiliser de processus XML Activator pour mettre à jour ou supprimer des documents possédant des éléments OTURN générés automatiquement.
Opération <Operation>, à moins que vous n'ayez spécifié une autre balise pour Operation dans Content Server. Action effectuée par Content Server sur l'objet de données. Cette valeur est définie sur AddOrReplace ou Delete. Si vous n'incluez pas cet élément dans un fichier XML, XML Activator définit par défaut sa valeur sur AddOrReplace.
Contenu <Content attribute="valeur"> Objet de données. Par défaut, attribute est défini sur Encoding, valeur est défini sur Base64 et le type de codage binaire pour le contenu est Base64. Vous pouvez changer ces paramètres en spécifiant d'autres paramètres pour les champs Balise de contenu, Attribut de balise, Valeur d'attribut de balise et Codage, dans cet ordre, dans l'onglet Infos spécifiques de la page Propriétés de chaque processus XML Activator. Toutes les balises HTML, XML ou SGML incluses dans la balise Content sont traitées comme élément du contenu.
Metadata <Metadata> Série de champs, sous forme de texte balisé, décrivant l'objet de données. Les champs de métadonnées agissent comme des attributs que des utilisateurs peuvent utiliser pour les recherches en mettant en correspondance des données avec des régions Content Server existantes ou avec des régions définies dans le fichier .tok du processus XML Activator. Le tableau suivant vous présente les champs de métadonnées requis, chacun d'entre eux correspondant à une région Content Server. Pour plus d'informations sur la définition de régions dans les fichiers .tok, reportez-vous à la documentation des développeurs Content Server.

Le tableau suivant décrit les champs de métadonnées qu'OpenText vous recommande d'inclure dans chaque fichier XML. Chaque champ correspond à une région Content Server. Si vous n'incluez pas ces champs, les utilisateurs Content Server ne pourront pas les rechercher dans Content Server.

Les noms de balise que vous utilisez doivent correspondre aux noms des champs de métadonnées, à moins que vous n'ayez spécifié d'autres mises en correspondance des balises dans Content Server dans le champ Liste des métadonnées du processus XML Activator, qui se trouve dans l'onglet Infos spécifiques de la page Propriétés du processus. Vous pouvez également inclure autant de champs de métadonnées supplémentaires que nécessaire en incluant les informations dans les balises dont les noms correspondent aux régions Content Server ou dont les noms sont mis en correspondance avec des régions dans le champ Liste des métadonnées du processus XML Activator. Pour plus d'informations sur la mise en correspondance de balises de métadonnées lors de l'ajout ou de la configuration d'un processus XML Activator, consultez l'aide en ligne administrateur Content Server.

Champs de métadonnées requis

Champ Description
OTName Nom de l'objet de données.
OTOwnerID Identificateur unique du propriétaire de l'objet de données.
OTLocation Emplacement d'origine de l'objet de données.
OTCreateDate Date de création de l'objet de données.
OTCreateTime Heure de création de l'objet de données.
OTModifyDate Date de la dernière modification de l'objet de données.
OTModifyTime Heure de la dernière modification de l'objet de données.
OTCreatedBy Numéro de nœud du créateur de l'objet de données.
OTCreatedByName Nom de connexion du créateur de l'objet de données.
OTCreatedByFullName Nom complet du créateur de l'objet de données.

Pour définir AddOrReplace comme valeur de Operation, structurez ainsi votre fichier XML :

<?xml version="x.x"?>
<Top_Level_Tag>
   <OTUrn>OTURN</OTUrn>
   <Operation>AddOrReplace</Operation>
   <Content attribute="value">Content data</Content>
   <Metadata>
      <OTName>Data object name</OTName>
      <OTOwnerID>Owner ID number</OTOwnerID>
      <OTLocation>Original location</OTLocation>
      <OTCreateDate>Creation date</OTCreateDate>
      <OTCreateTime>Creation time</OTCreateTime>
      <OTModifyDate>Date last modified</OTModifyDate>
      <OTModifyTime>Time last modified</OTModifyTime>
      <OTCreatedBy>
         Node number of creator
      </OTCreatedBy>
      <OTCreatedByName>
         Login name of creator
      </OTCreatedByName>
      <OTCreatedByFullName>
         Full name of creator
      </OTCreatedByFullName>
   </Metadata>
</Top_Level_Tag>

Pour définir Delete comme valeur de Operation, structurez ainsi votre fichier XML, en incluant uniquement OTURN et Operation. N'incluez pas de contenu ou de métadonnées car cette Operation supprime les données qui existent déjà dans l'index et est identifiée par son élément OTURN.

<?xml version="x.x"?>
<Top_Level_Tag>
   <OTUrn>OTURN</OTUrn>
   <Operation>Delete</Operation>
</Top_Level_Tag>
