Vous pouvez activer, configurer et programmer les événements suivants :

  • Producteur d'événements à minuit

    Exécute sur le système des événements réguliers qui n'ont pas leur propre programme d'activités et qui sont programmés pour s'exécuter une fois par jour.

    Il existe une API où les développeurs peuvent enregistrer les gestionnaires avec un événement afin d'exécuter ces gestionnaires à chaque exécution de l'événement. Dans une configuration système par défaut, ces gestionnaires comprennent :

    • TaskLate (Tâche en retard)

      qui génère des notifications pour les tâches qui sont en retard.

    • WorkflowLate (Workflow en retard)

      qui génère des notifications pour les workflows qui sont en retard.

    • WorkflowStepLate (Etape de workflow en retard)

      qui génère des notifications pour les étapes de workflow qui sont en retard.

    • DTreeNotifyPurge (Elimination de DTreeNotify)

      qui élimine les tableaux de notifications s'ils ne sont pas utilisés. Par exemple, si le système est uniquement un système de recherche, alors les tableaux de notifications ne sont pas utilisés.

    Par défaut, cet événement s'exécute tous les jours à minuit. Ce paramètre peut être modifié dans la section Programme d'activités.

  • Nouvel essai après l'échec de suppression du blob du fournisseur

    Si le stockage blob est utilisé pour un fournisseur de stockage, les données du blob peuvent être conservées après une suppression ; cet événement essaiera de supprimer toutes les données qui ne devraient plus être conservées.

    Il est également possible que les données du blob ne soient pas déplacées. Dans ce cas, cet événement essaiera de terminer toute demande de déplacement en suspens.

    Par défaut, cet événement s'exécute tous les jours à minuit. Ce paramètre peut être modifié dans la section Programme d'activités.

  • Glisser et déposer la notification d'éléments incomplets

    Informe les utilisateurs par e-mail lorsqu'ils font glisser et déposent un élément dans un recueil auquel est assignée une catégorie avec des attributs requis, mais qu'ils n'ont pas encore complété les informations d'attributs requises pour cet élément. En d'autres termes, cet élément envoie une notification lorsqu'un élément, qui a été glissé et déposé dans Content Server, n'a pas été entièrement ajouté à Content Server en raison de données incomplètes.

    Des notifications par e-mail sont envoyées aux utilisateurs ayant des éléments incomplets aux moments spécifiés dans le Programme d'activités, sauf si l'élément a été ajouté dans la limite de temps Retard d'exécution. Le paramètre Retard d'exécution donne aux utilisateurs un certain temps pour compléter les éléments une fois glissés et déposés dans un recueil, avant l'envoi d'une notification. Si, par exemple, le programme d'activités est défini pour rechercher des éléments incomplets à 13 h 00 et que le Retard d'exécution est défini sur 30 minutes, les éléments ajoutés entre 12 h 30 et 13 h 00 seront inclus dans la période de notification suivante dans le Programme d'activités.

    Par défaut, cet événement s'exécute toutes les cinq minutes, chaque jour. Ce paramètre peut être modifié dans la section Programme d'activités.

  • Vider les messages antérieurs

    Supprime les messages de notification de plus de trente jours.

    Cet événement s'exécute toutes les cinq minutes, chaque jour. Le programme n'est pas configurable pour cet événement.

    Les messages antérieurs seront vidés les jours et aux heures spécifiés sur le Programme d'activités du producteur d'événements à minuit.

  • Faire expirer les mots de passe

    Désactive les comptes des utilisateurs dont les mots de passe ont expiré. Ceci garantit qu'un nouveau mot de passe ne pourra pas être spécifié lors de la prochaine tentative de connexion à Content Server de l'utilisateur.

    Par défaut, cet événement s'exécute tous les jours à minuit. Ce paramètre peut être modifié dans la section Programme d'activités.

  • Contrôle de l'expiration du mot de passe

    Assure le suivi du statut d'expiration du mot de passe de tous les utilisateurs et informe les utilisateurs par e-mail lorsque leur mot de passe est sur le point d'expirer. Vous saisissez le nombre de jours avant l'expiration du mot de passe où vous souhaitez que la première notification par e-mail soit envoyée. Vous saisissez également le nombre de jours durant lesquels vous souhaitez que les notifications par e-mail soient exécutées.

    Par défaut, cet événement s'exécute tous les jours à minuit. Le programme n'est pas configurable pour cet événement.

  • Notification d'échec de connexion

    Informe l'administrateur lorsque le nombre d'échecs de tentatives de connexion excède un seuil configurable dans un délai lui aussi configurable.

    Par défaut, cet événement s'exécute tous les jours à minuit. Ce paramètre peut être modifié dans la section Programme d'activités.

  • Agent client 5 minutes

    Cet événement est identique à celui du producteur d'événements à minuit, excepté qu'il s'exécute conformément à un programme différent. Cet événement est utilisé lorsque vous avez besoin d'exécuter des tâches plus d'une fois par jour.

    Il existe une API où les développeurs peuvent enregistrer les gestionnaires avec un événement afin d'exécuter ces gestionnaires à chaque exécution de l'événement. Dans une configuration système par défaut, ces gestionnaires comprennent :

    • ClearCache (Vider cache)

      qui vide les entrées antérieures de la table LLCache.

    • DistributedAgent Controller (Contrôleur de l'agent distribué)

      qui contrôle les files d'attentes du travailleur pour les tâches hors service et les marque pour un nouvel essai ou une annulation, selon le cas.

    • PollClosed (Sondage fermé)

      qui rend compte des sondages qui ont été fermés depuis la dernière exécution.

    Par défaut, cet événement s'exécute toutes les cinq minutes, chaque jour. Ce paramètre peut être modifié dans la section Programme d'activités.

  • Transférer les événements de notification

    Déplace les événements de NotifyEvents à LLEventQueue pour leur traitement.

    Par défaut, cet événement s'exécute toutes les cinq minutes, chaque jour. Ce paramètre peut être modifié dans la section Programme d'activités.

  • Processeur d'événements de client

    Les développeurs ne doivent plus sous-classer ce processus car il s'agit d'un agent obsolète.

  • Processeur d'événements du nœud

    Convertit les événements de nœud individuels en événements de message pour tous les utilisateurs que cette combinaison nœud/événement intéresse. Vous pouvez définir le nombre d'instances du processeur d'événements d'objet exécutées en parallèle.

    Par défaut, cet événement s'exécute toutes les cinq minutes, chaque jour. Ce paramètre peut être modifié dans la section Programme d'activités.

    Une option supplémentaire permettant d'activer ou de désactiver les vérifications d'autorisation pour les notifications des groupes et des rôles est disponible. Le paramètre est activé par défaut, ce qui signifie que les autorisations sont vérifiées lors de chaque activité programmée. Si le paramètre est désactivé, la vérification d'autorisation supplémentaire n'a lieu que lorsque l'utilisateur intéressé est un utilisateur individuel. La vérification est ignorée lorsque l'utilisateur intéressé est un rôle (par exemple, pour les projets ou les communautés).

  • Evénements de recherche de sort final

    Cet événement est utilisé pour activer les recherches de sort final dans Records Management. Cet événement est désactivé par défaut.

  • Attentes avec recherche

    Lorsque cet événement est activé, les requêtes d'attente dans Records Management s'exécutent automatiquement en fonction des jours et des heures que vous spécifiez dans la section Programme d'activités.

  • Règles définies pour l'habilitation de sécurité

    Lorsque cet événement est activé, les règles définies pour l'habilitation de sécurité sont exécutées les jours et aux heures que vous spécifiez pour l'agent, tant que la valeur Date d'exécution de l'élément a été atteinte. La valeur Date d'exécution est spécifiée lorsque la règle définie est créée. Lorsque les règles définies sont exécutées, les marquages supplémentaires et les niveaux d'habilitation de sécurité de l'élément sont mis à jour selon la règle définie.

 

Note :
Les utilisateurs peuvent personnaliser les trois programmes par défaut lorsqu'ils configurent leur propre programme de notification personnel.
