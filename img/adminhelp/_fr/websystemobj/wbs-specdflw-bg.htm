Une fois que vous avez créé un flux de données, vous pouvez gérer ses IPools et processus de flux de données. Vous pouvez lancer, arrêter, vider, tester, resynchroniser et déplacer les flux de données et leurs composants. Pour plus d'informations sur les flux de données, reportez-vous à la rubrique Gestion des flux de données.

Démarrage et arrêt des flux de données

Lorsque vous gérez un flux de données, vous pouvez arrêter ou lancer tous les processus de flux de données en même temps. Certaines tâches de gestion des flux de données (par exemple, le test ou l'élimination d'un flux de données) peuvent nécessiter leur suspension. Si un flux de données contient un ou plusieurs processus devenus inactifs dans le cadre de leur fonctionnement normal, vous souhaitez peut-être le suspendre pour le reprendre ensuite. Par exemple, le processus Directory Walker du flux de données d'aide devient inactif après avoir parcouru les répertoires d'aide. Si vous ajoutez un module Content Server (tels que Content Server Classifications), vous devez mettre à jour l'index de l'aide pour que les fichiers d'aide associés au nouveau module soient ajoutés à l'index de l'aide.

Définition des règles de contrôle

Vous pouvez également définir des règles de contrôle pour les IPools et les processus des flux de données. Les règles de contrôle permettent de définir des options telles que le nombre de messages IPool associés à un processus particulier dans un flux de données. Pour plus d'informations sur la définition des règles de contrôle, reportez-vous à la rubrique Ajout de règles de contrôle de flux de données.

Test des flux de données

Pour que les données circulent convenablement dans un flux de données, les chemins spécifiés pour les IPools de lecture et d'écriture de chacun de leurs processus doivent être valables et précis. Si vous avez l'impression que les données ne circulent pas correctement dans un flux de données, vous pouvez le tester. Lors de ce test, Content Server suspend le flux de données, supprime le contenu de tous les répertoires iPool, puis essaie d'envoyer un message test via le flux de données. Si Content Server rencontre une erreur, un message d'erreur s'affiche. S'il n'y a pas d'erreurs, Content Server affiche le contenu du dossier préfixe_processus Dossier source de données. Pour réparer un flux de données défectueux, cliquez sur le bouton Informations de l'IPool de l'onglet Spécifique de la page Propriétés de chaque processus et assurez-vous que les chemins affichés dans les champs Lire et Ecrire sont valables et précis, puis recommencez le test.

Elimination des flux de données

Si l'un des processus d'un flux de données semble avoir des difficultés à traiter le volume de données en cours d'écriture dans son IPool source, vous pouvez éliminer le flux de données. L'élimination d'un flux de données le suspend et supprime tous les messages IPool en attente à l'intérieur.

Resynchronisation des flux de données

Si un ou plusieurs répertoires IPool ont été supprimés par inadvertance dans un flux de données, vous pouvez les recréer en resynchronisant le flux de données. Lors de la resynchronisation, Content Server lit les informations stockées dans la base de données relatives à l'emplacement des répertoires iPool, puis les recrée ou les répare si nécessaire. Une fois que vous avez resynchronisé un flux de données, testez-le en envoyant un message test via les IPools.

Maintenance des composants individuels d'un flux de données

Vous pouvez également gérer les composants individuels d'un flux de données. Vous pouvez déplacer les iPools et les processus d'un flux de données, lancer ou arrêter des processus individuels, configurer les options de lancement ou d'arrêt des processus ou resynchroniser les processus avec la base de données Content Server.

Resynchronisation des processus de flux de données

Le serveur stocke des informations sur tous les processus de flux de données qu'il contrôle dans la base de données Content Server. Chaque serveur d'administration gère des informations sur les processus de flux de données qu'il gère dans un fichier otadmin.cfg. Les informations figurant dans le fichier otadmin.cfg d'un serveur d'administration doivent toujours correspondre aux informations correspondantes dans la base de données Content Server. Dans les cas suivants, ces informations peuvent diverger :

  • Les informations relatives à un processus de flux de données particulier dans le fichier otadmin.cfg d'un serveur d'administration enregistré sont corrompues ou ont été supprimées.

  • Les informations relatives à un processus de flux de données particulier dans la base de données Content Server ont été modifiées et les modifications correspondantes n'ont pas pu être effectuées dans le fichier otadmin.cfg du serveur d'administration qui gère le processus. Cela peut arriver si les commandes SQL qui modifient ces informations sont envoyées directement à la base de données Content Server. OpenText vous recommande vivement de ne pas envoyer de commandes SQL directement à la base de données Content Server.

Si les informations d'un processus de flux de données particulier stockées dans la base de données Content Server vous semblent inexactes ou manquantes dans le fichier otadmin.cfg du serveur d'administration qui gère le processus, vous pouvez resynchroniser le processus. Lors de la resynchronisation d'un processus, les informations relatives au processus enregistrées dans la base de données Content Server sont écrites dans le fichier otadmin.cfg du serveur d'administration qui gère le processus.

 

Note :
Les informations relatives au processus Distributeur de mise à jour sont également enregistrées dans le fichier search.ini. Le fichier search.ini est un fichier de configuration qui contient les paramètres des composants des systèmes d'indexation et de recherche (par exemple, partitions, moteurs d'indexation et moteurs de recherche). La resynchronisation du processus Distributeur de mise à jour écrit les informations relatives au processus dans les fichiers otadmin.cfg et search.ini.
