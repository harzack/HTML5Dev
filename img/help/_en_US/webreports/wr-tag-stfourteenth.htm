This section details the sub-tags: PARSECSV, PATCHANGE, PATFIND, PERMACTION, PERMCHECK, PERMINFO, PERMTOASSOC, PLUCK, PLURALIZE, POCOPY, POINFO and PREV.

PARSECSV

Syntax

PARSECSV

Description

This sub-tag parses a string with comma-separated values and returns a structured RecArray.

Example

For example:

[LL_REPTAG_'A,B,C' PARSECSV /]

Returns:

V{<'Column1','Column2','Column3'><'A','B','C'>}

Related Information

For more information, see FOR.

PATCHANGE

Syntax

PATCHANGE:<find pattern>:<new pattern>

or

PATCHANGE:<find pattern>:<new pattern>:<ignoreCase?>

Description

Returns a modified version of the main tag based on the <find pattern> and <new pattern> specified.

The <find pattern> is specified with a series of special operators as per the table below. This is a simplified and slightly modified subset of UNIX's regular expression pattern matching. For Content Server developers this syntax is based on the Oscript PATFIND and PATCHANGE builtins. This is almost identical to the PATFIND syntax but includes the ability to specify parts of the <find pattern> which can be referenced for substitution in the <new pattern>. If no match if found, the original tag contents is returned unchanged.

The <new pattern> can also include some special characters which are also shown below. The <new pattern> is normally a fixed pattern of characters but any characters to be retained from the <find pattern> are enclosed in the <...> characters and then inserted in the <new pattern> using the pound, #, symbol. By default this sub-tag ignores the case of characters when matching. Specifying FALSE as a third parameter causes the change operation to be case sensitive.

Parameters

Table 1-36: Find Pattern Special Characters

Character
Description
c
Match exact character.c is any character here.
@c
Match escaped character, such as @%, @[, @*.
?
Match any character except end of line.
%
Match beginning of line.
$
Match end of line. Null string before end of line.
[...]
Match character class. In other words, match any of the specified characters, such as [abc] or [a-z].
[!...]
Match negated character class. In other words, match all but these characters, such as [! ] for no spaces.
*
Match zero or more occurrences of the previous element, such as [0-9]* for zero or more digits.
+
Match one or more occurrences of the previous element, such as [A-Za-z]+ for one or more letters.
<...>
Brackets tagged material which is specially extracted for later reference or use.

.
Table 1-37: New Pattern Special Characters

Character
Description
&
Ditto (Whole String).
@c
Escaped character, such as @&.
#n
Tagged substring insertion.

Example

This example of the PATCHANGE sub-tag assumes that the associated tag contains the string: “This is a test sentence 123, end”

PATCHANGE:"?* <t?+t> *sen[!0-9]+<[0-9]+>?*":"Word: #1; Number: #2"

would return: Word: test; Number: 123

In the example above:

  1. Any characters that are matched, but not enclosed in angled brackets, <>, will not be included in the replacement.

  2. <t?+t>: finds a match with the word “test”. Because it is the first “tagged substring”, this word is substituted in place of #1 in the replace string.

  3. <[0-9]+>: finds a match with the numbers “123”. Because it is the second “tagged substring”, these characters are substituted in place of #2 in the replace string.

  4. The part of the find pattern: sen[!0-9]+ finds the characters “sen” followed by 1 or more characters that are NOT numbers.

Related Information

For more detailed information about PATFIND or PATCHANGE the Content Server SDK documentation should be consulted.

PATFIND

Syntax

PATFIND:<find pattern>

or

PATFIND:<find pattern>:<ignoreCase?>

Description

Returns either TRUE or FALSE based on whether the specified pattern is found within the string returned by the main tag. The <find pattern>is specified with a series of special operators as per the table below. This is a simplified and slightly modified subset of UNIX's regular expression pattern matching. For Content Server developers this syntax is based on the Oscript PATFIND and PATCHANGE builtins. By default, this sub-tag ignores the case of characters when matching. Specifying FALSE as a second parameter causes the find operation to be case sensitive.

Parameters

Table 1-38: Find Pattern Special Characters

Character
Description
c
Match exact character.c is any character here.
@c
Match escaped character, such as @%, @[, @*.
?
Match any character except end of line.
%
Match beginning of line.
$
Match end of line. Null string before end of line.
[...]
Match character class. Match any one of these characters, such as [abc] or [a-z].
[!...]
Match negated character class. In other words, match all but these characters, such as [! ] for no spaces.
*
Match zero or more occurrences of the previous element, such as [0-9]* for zero or more digits.
+
Match one or more occurrences of the previous element, such as [A-Za-z]+ for one or more letters.

Example

The following example of the PATFIND sub-tag assumes that the associated tag contains the string, “This is a test sentence 123, end”:

PATFIND:"sen?+ *[0-9]+"

would return “TRUE”.

This pattern details that:

  1. the specific characters, “sen”,

  2. followed by one or more characters as directed by “?+”,

  3. followed by zero or more spaces as directed by “ *”,

  4. followed by one or more digits as directed by “[0-9]+”,

As such, the tag will find a match on the string “sentence 123”, and therefore the tag returns TRUE.

Related Information

For more information about PATFIND or PATCHANGE see the Content Server SDK documentation.

PERMACTION

Syntax

PERMACTION:<operation>

Description

This sub-tag provides the flexibility to add, remove, and modify permissions of content within Content Server. Because of the powerful nature of this sub-tag, its effects should be carefully understood before usage. The sub-tag behaves like the other WebReports sub-tags in that you cannot modify the permissions of nodes to which you do not have permissions through the regular interface. The key difference is that using WebReports allows these changes to be automated so they can be done much faster than through the interface. Content Server supports 3 permission models:

  • the standard permission model (SPM), for example the Content Server objects such as documents and folders

  • the project permission model (PPM), for example, projects

  • and the work items permission model (WPM), for example, discussions, channels, or task lists.

The PERMACTION sub-tag provides the functionality to update all 3 permission models. The tables below describes how to use the PERMACTION sub-tag and the object(s) it affects.

Parameters

In the tables below, the following acronyms are used:

  • ACL: the access control list that defines those users or groups who have been granted permission to access a Content Server object.

  • SPM: the standard permission model that includes documents and folders.

  • PPM: the project permission model that includes projects.

  • WPM: the work items permission model that includes discussions, channels, and tasks.

Table 1-39: PERMACTION:ACL:<userID>

Operation
Description
PERMACTION:ACL:<userID>:REMOVE
Removes the user or group specified by <userID> from a Content Server object's permission ACL. This operation is supported by SPM and WPM objects.
PERMACTION:ACL:<userID>:UPDATE:<perms>
Updates the permissions specified by <perms> for the user or group specified by <userID> in a Content Server object's permission ACL. If the user or group specified is not currently in the ACL, they will be added. This operation is supported by SPM and WPM objects.

PERMACTION:OWNER:UPDATE:<perms>
Updates the permissions specified by <perms> for the current owner in the Content Server object's default access. This operation is supported by SPM and WPM objects.
PERMACTION:OWNER:CHANGE:<userID>:<perms>
Sets the owner section of the default access for a Content Server object's permissions to the user specified by <userID> with the permissions specified by <perms>. This operation is supported by SPM and WPM objects.
PERMACTION:OWNER:REMOVE
Removes the current owner from the default access section of permissions for the Content Server object. This will set the OWNER section of default access to “No Owner”. This operation is supported by SPM and WPM objects.

Sets the owner group section of the default access for a Content Server object's permissions to the group specified by <userID> with the permissions specified by <perms>. This operation is supported by SPM and WPM objects.
PERMACTION:OWNERGROUP:REMOVE
Removes the current owner group from the default access section of permissions for the Content Server object. This will set the owner group section of default access to “No Owner Group”. This operation is supported by SPM and WPM objects.

Updates the Role specified by <role>, for the user or group specified by <userID>, in the Project's Participant page. If the user or group specified is not a Project Participant, they will be added. This operation is only supported by PPM objects.

This sub-tag expects the data id of the content in Content Server the user wants to perform permission actions upon, to be specified in the main tag. For SPM objects, for example documents or folders, there are two ways to specify permissions for any action that requires <perms>:

  1. You can use a decimal number representing a binary bit mask for Content Server permissions. While some developers may be familiar with this, it is too much to display here.

  2. You can also specify a list of permissions using any combination of the following keywords:

    Table 1-44: Permissions Keywords

    Keyword
    Permission Affected
    SEE
    See
    SEECONTENTS
    See Contents
    MODIFY
    Modify
    EDITATTRS
    Edit Attributes
    RESERVE
    Reserve or Checkout
    DELETEVERS
    Delete Versions
    DELETE
    Delete the Object
    EDITPERMS
    Edit Permissions
    ADDITEMS
    Add Items

Note that the permissions specified in this sub-tag follow the permissions model of the Content Server interface, meaning that selecting just one permission will also set the other required permissions. For example, choosing MODIFY will set the “See”, “See Contents” and “Modify” permissions for an object.

To specify permissions for WPM objects, for example discussions, channels or task lists, the following keywords can be substituted for <perms> in any of the applicable PERMACTION operations:

  • NONE

  • READ

  • WRITE

  • ADMINISTER

To specify permissions for PPM objects, for example Projects, the following table displays the three types of roles that can be substituted for <<role>> in the applicable PERMACTION operations:

  • COORD: indicates a Coordinator.

  • MEMBER

  • GUEST

For the scenario where PUBLIC permissions can be updated for PPM objects, PERMACTION:PUBLIC:UPDATE:<perms>, the acceptable values for <perms> in this case is ENABLED or DISABLED.

Example

Example

If a user creating a WebReport wants to update the permissions for a user with a user ID of 1234 in the ACL of a document, in this case DATAID, with the permissions of See and See Contents, they could use either:

[LL_REPTAG=DATAID PERMACTION:ACL:1234:UPDATE:"SEE,SEECONTENTS" /]

Which is using the list of keywords, SEE,SEECONTENTS to specify the permissions to update, or, the user could:

[LL_REPTAG=DATAID PERMACTION:ACL:1234:UPDATE:36995 /]

Which is specifying a decimal representation of the binary bit mask, 36995, for See and See Contents.

Example

If a user creating a WebReport wants to change the Owner of a task list to the Admin user, with a userid of 1000, and give them Administer access, they can use the following tag:

[LL_REPTAG=DATAID PERMACTION:OWNER:CHANGE:1000:ADMINISTER /]

Where DATAID returns the dataID of the task list and ADMINISTER is a keyword for WPM objects.

Example

If a user creating a WebReport wants to set the Admin user, with a userid of 1000, in a Project to a Member role, the following tag can be used:

[LL_REPTAG=DATAID PERMACTION:MEMBER:UPDATE:1000 /]

Where DATAID returns the dataID of the project and MEMBER is a keyword for PPM objects.

PERMCHECK

Syntax

PERMCHECK:<parm>

or

PERMCHECK:<parm>:<user/group ID>

Description

Checks if the current, or a specified, user has permission to perform the specified action for that object. Returns TRUE or FALSE based on the result of this verification.

 

  1. When this tag is used, the value that would normally be returned by the main tag is replaced by either TRUE or FALSE.

  2. This tag is only valid when used with a tag that normally returns a valid Content Server object id. For example, IDs, DATAIDs, and VOLUMEIDs. An invalid report tag will cause this sub-tag to return FALSE.

  3. The specified actions map to Content Server permissions as shown in the chart below. Examples of actions include see, edit, and delete. To use this chart:

    1. Find the highest level of Content Server permission that the user is expected to have, and that will therefore return TRUE from this tag.

    2. Use the setting listed under Permcheck Action.

 

Parameters

Permcheck Action
Forms
DTree Objects
    NONE
SEE   SEE
SEECONTENTS SEE SEE CONTENTS
EDIT EDIT STATIONERY MODIFY
EDIT SQL SUBMIT EDIT ATTRIBUTES
EDIT   DELETE VERSIONS
EDIT   RESERVE
EDIT   ADD ITEMS
MODIFY   MODIFY
EDITATTRIBUTES   EDIT ATTRIBUTES
RESERVE   RESERVE
DELETE ADMINISTER DELETE
DELETE   EDIT PERMISSIONS
DELETEVERSIONS   DELETE VERSIONS
EDITPERMISSIONS   EDIT PERMISSIONS
ADDITEMS   ADD ITEMS

Example

If the EDIT action has been specified against a form DataID, then the tag will only return TRUE if the user has at least “Edit Stationery” permission for the associated form.

To make sure that a delete button is only visible for users who have “Administer” permission to an associated form, the following WebReport declaration could be used. This assumes that form Dataid is being returned in the underlying report data:

[LL_WEBREPORT_IF "[LL_REPTAG=DATAID PERMCHECK:DELETE /]" == "TRUE" /]
   <INPUT TYPE=BUTTON NAME="del" VALUE="Delete" ONCLICK="deleteItem()">
[LL_WEBREPORT_ENDIF /]

This could also be done using JavaScript code as below; however, this code will still be visible if a user views the source HTML from their browser:

If ( "[LL_REPTAG=DATAID PERMCHECK:DELETE /]" == "TRUE" ) {
   document.write('<INPUT TYPE=BUTTON NAME="delbutton" VALUE="Delete" ONCLICK="deleteItem();">');
};

PERMINFO

Syntax

PERMINFO:<format>

Description

For a valid node ID, this sub-tag will a return a list of Assocs containing permission data for the node according to the format parameter that has been provided. Can be chained with the LIST and ASSOC sub-tags to return more specific information. Only returns data if the user has Edit Permissions on the node, otherwise an empty string will be returned.

Parameters

Format
Description
PERMINFO
Returns a list containing separate Assocs for the Owner, Owner Group, Public Access and any Assigned Access entries.
•

The first Assoc will contain the permission data for the Owner.

•

The second Assoc will contain the permission data for the Owner Group.

•

The third Assoc will contain the permission data for Public Access.

•

If there are any Assigned Access entries, then there will be a separate Assoc for each one from the fourth list entry onwards.

PERMINFO:OWNER
Returns a list containing an Assoc containing the permission data for the Owner.
PERMINFO:OWNERGROUP
Returns a list containing an Assoc containing the permission data for the Owner Group.
PERMINFO:PUBLICACCESS
Returns a list containing an Assoc containing the permission data for Public Access.
PERMINFO:ASSIGNEDACCESS
Returns a list containing separate Assocs for each entry in Assigned Access.

Example

PERMINFO LIST:1 ASSOC:NAME

Will return the name of the node Owner.

Example

PERMINFO:PUBLICACCESS LIST:1 ASSOC:PERMASSOC ASSOC:RESERVE

Will return TRUE or FALSE based on whether Public Access has Reserve permission on the node.

Related Information

For more information, see the LIST and ASSOC sub-tags.

PERMTOASSOC

Syntax

PERMTOASSOC

Description

This sub-tag converts a permission bitmask into an Assoc for easier reading.

Example

The following tag:

[LL_REPTAG_'36995' PERMTOASSOC /]

Will return:

A<1,?,'Create'=false,'Delete'=false,'DeleteVer'=false,'EditAttr'=false,'EditPerm'=false,'Modify'=false,'Reserve'=false,'See'=true,'SeeContent'=true>.

Related Information

For more information, see the PERMACTION and ASSOC sub-tags.

PLUCK

Syntax

PLUCK<field>

Description

The PLUCK sub-tag is used to extract a value from a collection of Records or Assocs, and returns the values as a List. The main tag must be a RecArray, or List of Assocs, or List of Records.

Example

The following tag will extract all the NAME values from the data source:

[LL_REPTAG_DATASOURCE PLUCK:NAME /]

Will return:

{'Show All [Type] Items','My Tasks Due Next Week','All Tasks Due Next Week', ... }

Related Information

For more information, see the LIST, RECORD, and ASSOC sub-tags.

PLURALIZE

Syntax

PLURALIZE<suffix>

Description

Returns a plural suffix if the main tag is not 1. By default, this suffix is “s”.

Example

Consider the following tag:

[LL_REPTAG_DATASOURCE LENGTH /] row[LL_REPTAG_DATASOURCE LENGTH PLURALIZE /]

If [LL_REPTAG_DATASOURCE LENGTH /] is 1, the output will be “1 row”. If [LL_REPTAG_DATASOURCE LENGTH /] is 2, the output will be “2 rows”.
