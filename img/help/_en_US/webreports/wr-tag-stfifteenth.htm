This section details the sub-tags: RANGE, REAL, REC, RECARRAY, RECORD, REDIRECT, REPLACE, REVERSE, RMACTION, RMCLASS, RMHOLDACTION, RMHOLDINFO, RMINFO, RMPERMCHECK, RMTYPE, ROUND, RPAD, RSI, RSISCHEDULE, RUNSHELL, RUNSQL, RUNSWR and RUNTEMPLATE.

RANGE

Syntax

RANGE:<start index>

or

RANGE:<start index>:<end index>

Description

For any list formatted as: {'item1','item2'}, this sub-tag returns a range from the original list based on a specified start index and an optional end index. This sub-tag is identical to the SLICE sub-tag but only works with LISTs. All indices count starting at one (1) so the slice will include all characters starting from, and including, the <start index> and concluding with, and including, the <end index>, if one is specified. If <end index> is omitted, or exceeds the total length of the string, then the remainder of the string is used. A LIST follows the Content Server LIST convention where the first character is a {, the last character is a }, and each element is separated by a comma. The output of this sub-tag is a list in the same format. A LIST can be de-referenced using the LIST sub-tag.

Example

These different examples of the RANGE sub-tag show the different results that would be produced, assuming the associated tag contained the following list: {'Apple','Pear','Orange','Mango'}

RANGE:3 would return: {'Orange','Mango'}

RANGE:2:3 would return: {'Pear','Orange'}

RANGE:3:100 would return: {'Orange','Mango'}

RANGE:2:3 LIST:1 would return: 'Pear'

REC

Syntax

REC

Description

The REC sub-tag operates on a DataID, and returns the related row from the WebNodes view. The sub-tag also accepts a DTREE parameter, that returns the DTree record for the provided DataID.

Example

[LL_REPTAG_'2000' REC /]

Returns the WebNodes record for the Enterprise:

R<'OWNERID'=-2000,'PARENTID'=-1,'DATAID'=2000,'NAME'='Enterprise', ... >

RECARRAY

Syntax

RECARRAY:<rowIndex>:<colIndex>

or

RECARRAY:<rowIndex>:<fieldName>

Description

Allows indexing of the Content Server RECARRAY data type. If the column extracted from the data source is a RECARRAY, a Content Server data structure consisting of a LIST of RECORDs, the value of the field specified by <rowIndex> and <colIndex> or <fieldName> is returned, and only that value.

 

Note:
This sub-tag can be used repetitively and/or in combination with other sub-tags, such as ASSOC and LIST, to access information in complex data structures.

 

Related Information

For more information, see ASSOC, LIST and RECORD.

RECORD

Syntax

RECORD:<index>

or

RECORD:<fieldName>

Description

Allows indexing of the Content Server RECORD data type. If the column extracted from a data source is a RECORD, in other words a Content Server data structure, the value of the field specified by <index> or <fieldName> is returned, and only that value.

The <index> argument will only work if the data does not contain commas (,) or equals signs (=). If the data contains commas or equals signs, OpenText recommends that you use <fieldName>.

 

Note:
This sub-tag can be used repetitively and/or in combination with other sub-tags such as ASSOC and LIST to access information in complex data structures.

 

Related Information

See also: ASSOC, LIST and RECARRAY.

REDIRECT

Syntax

REDIRECT

Description

REPLACE

Syntax

REPLACE:<parm1>:<parm2>

Description

Performs a “replace all” action on the items of data pulled from the data source. All occurrences of the <parm1> string are replaced by <parm2>.

Example

[LL_REPTAG=PRIORITY REPLACE:"3":"Low" /] would result in all occurrences of the number “3” in the Priority field being replaced with the word “Low”.

 

Note:
There can be no spaces between REPLACE, <parm1> and <parm2>. Use only the colon separator.

 

REVERSE

Syntax

REVERSE

Description

Accepts a list and reverses it.

Example

[LL_REPTAG_'{1,3,2}' REVERSE /] returns {2,3,1}.

RMCLASS

Syntax

RMCLASS:<parm>

Description

Provides metadata information unique to RM Classification objects. Requires the dataid of the RM Classification itself to be returned from the main tag. If you need to access details of an RM Classification from the node to which it is applied the RMINFO sub-tag can be used: [LL_REPTAG=DATAID RMINFO:CLASSID RMCLASS:<parameter> /].

RSI details can be accessed in a similar fashion with the RSI sub-tag. For example, [LL_REPTAG=DATAID RMINFO:CLASSID RMCLASS:RSI RSI:URL /] would get the URL to open an RSI based purely on the dataid of the node to which the RSI/RM Classification is applied.

It should be noted that an RM Classification is just a standard Content Server node type and therefore basic metadata associated with the node is available using the NODEINFO sub-tag. For example, [LL_REPTAG=DATAID NODEINFO:CREATEDATE /] will return the date the RM Classification was created. This differs from RMCLASS which returns metadata that is unique to the RM Classification object subtype and only found on the specific tab of such objects. If the Records Management module is not installed this sub-tag returns nothing for all options.

RMHOLDACTION

Syntax

RMHOLDACTION:<parms>

Description

Provides the ability to perform various actions to do with Holds, such as applying and removing Holds, creating new Holds, and creating new Hold Types.

If the Records Management module is not installed, this sub-tag returns nothing for all options.

Parameters

  • RMHOLDACTION:APPLY:<Hold ID or Name>

    Applies a Hold to the node specified in the main tag.

  • RMHOLDACTION:REMOVE:<Hold ID or Name>

    Removes a Hold from the node specified in the main tag.

  • RMHOLDACTION:CREATE:HOLD:<parms>

    Creates a new Hold. The data in the main tag is ignored. The following parameters can be used:

    Parameter Name
    Details
    Hold Name
    Mandatory parameter. Will be converted to upper-case
    Date Applied
    Mandatory parameter. Must be a valid date string, for example “yyyy-mm-dd”.
    Hold Type
    Mandatory parameter. Will be validated against existing Hold Types
    Alternate Hold ID
    Optional parameter.
    Comment
    Optional parameter.
    Date to Suspend
    Optional parameter. Must be a valid date string, if defined, for example “yyyy-mm-dd”.

    Example

    The following will create a Hold called “Hold 1”, with a Date Applied of “1st March 2012”, of type “Hold Type 1”:

    [LL_REPTAG_'123456' RMHOLDACTION:CREATE:HOLD:"Hold 1":"2012-03-01":"Hold Type 1" /]

    Example

    The second example will create a Hold called “Hold 2”, with a Date Applied of “1st March 2012”, of type “Hold Type 2”, with a blank value for “Alternate Hold ID”, a comment of “a comment” and a Date To Suspend of “15th March 2012”:

    [LL_REPTAG_'123456' RMHOLDACTION:CREATE:HOLD:"Hold 2":"2012-03-01":"Hold Type 1":"":"a comment":"2012-03-15" /]

  • RMHOLDACTION:CREATE:TYPE:<parms>

    Creates a new Hold Type. The following parameters can be used:

    Parameter Name
    Details
    Code
    Mandatory parameter. Will be converted to upper-case.
    Default Period
    Mandatory parameter. Must be an integer.
    Description
    Optional parameter. Will be converted to upper-case

    Example

    This first example will create a Hold Type called “HOLD TYPE 1” with a default period of 30:

    [LL_REPTAG_'123456' RMHOLDACTION:CREATE:TYPE:"Hold Type 1":30 /]

    Example

    The second example will create a Hold called “HOLD TYPE 2”, with a default period of 30 and a description of “HOLD TYPE 2 DESCRIPTION”.

    [LL_REPTAG_'123456' RMHOLDACTION:CREATE:TYPE:"Hold Type 2":30:"Hold Type 2 Description" /]

Related Information

See also RMACTION, RMCLASS, RMHOLDINFO, RMINFO, RMPERMCHECK, RMTYPE, RSI and RSISCHEDULE for other Records Management functionality.

ROUND

Syntax

ROUND:<round number>

Description

Rounds a numeric value by the specified ROUND parameter. If no ROUND parameter is specified, it defaults to 2.

The ROUND parameter must be an integer or real number. The integer or real number must be greater than, or equal to, 0. An error occurs if the tag does not contain a valid integer or real number. Zeroes are padded to the specified ROUND parameter if no decimal portion is present. See the second example below.

Example

  1. 25.479 ROUND:2 would return: “25.48”.

  2. 45 ROUND:3 would return: “45.000”. Note that zeroes are padded.

  3. 50.99 ROUND:0 would return: “51”.

  4. -4.545 ROUND:1 would return: “-4.5”.

RPAD

Syntax

RPAD:<pad char>:<width>

Description

Add padding characters, defined by <pad char>, to the right of existing data. Add enough characters to fill the field up to the specified <width>. If there is a mantissa, then only the width of the number following the decimal point is used.

Example

For example, given a data tag returning the number 123.456, the following [LL_REPTAG_1 RPAD:9:5 /] tag will return “123.45699”.

Given a data tag returning the number 123, the following [LL_REPTAG_1 RPAD:9:5 /] tag will return “12399”.

RUNSHELL

Syntax

RUNSHELL

Description

This sub-tag allows a server-side shell script to be executed on the server, and returns the output from the script.

Scripts must exist in a sub-directory below the “templates” folder under the WebReports module directory, and can have a .cmd, .bat, or .py extension.

RUNSHELL supports passing parameters to the script. The parameters must be separated from the script name by a space. For example, [LL_REPTAG_'scripts/myscript.py parm' RUNSHELL /]

Example

For example, a Python script with the following:

# contents of scripts/myscript.py
print 'Hello World!'

can be executed with:

[LL_REPTAG_'scripts/myscript.py' RUNSHELL /]

This would return 'Hello World!' to the report.

Related Information

See also RUNSQL, RUNTEMPLATE, and SETDATASOURCE.

RUNSQL

Syntax

RUNSQL

Description

This sub-tag is used to run a LiveReport and returns the results as a RecArray. The main tag must contain the DataID of a LiveReport or the file system data source name. Parameters passed to RUNSQL are interpreted as &inputLabel1, &inputLabel2, and so on.

If the LiveReport runs an UPDATE or DELETE statement then the RecArray will contain a single record containing <rowCount>, that will be populated by the number of rows updated or deleted.

Example

For example, [LL_REPTAG_'9999' RUNSQL:"JohnDoe" /] executes the LiveReport with DataID 9999 and passes in the parameter &inputLabel1=JohnDoe.

RUNSWR

Syntax

RUNSWR

Description

This sub-tag provides a sub-tag syntax for calling a sub-WebReport.

Example

The following tag demonstrates how a sub-WebReport is normally called with the [LL_WEBREPORT_SUBWEBREPORT /] tag:

[LL_WEBREPORT_SUBWEBREPORT NODEID:68873 PARM:inputLabel1:299 PARM:myStatus:"awaiting review" /]

This can be rewritten with the RUNSWR sub-tag as follows:

[LL_REPTAG_'68873' RUNSWR:inputLabel1:299:myStatus:"awaiting review" /]

Both approaches produce the same result, except the second approach allows further sub-tags to be added to the tag. For example:

[LL_REPTAG_'68873' RUNSWR:inputLabel1:299:myStatus:"awaiting review" DECODE:0:"false":1:"true" /]

If the destination of the sub-WebReport is a Content Server Node, then the nodeID can be returned to the calling WebReport by adding the SHOWNODEID argument to the end of the RUNSWR call. For example:

[LL_REPTAG_$mySubReport RUNSWR:SHOWNODEID /]
[LL_REPTAG_$mySubReport RUNSWR:inputLabel1:299:myStatus:"awaiting review":SHOWNODEID /]

Related Information

For more information, see Calling Sub-WebReports, Using Sub-WebReports and SUBWEBREPORT.

RUNTEMPLATE

Syntax

RUNTEMPLATE

Description

Calling RUNTEMPLATE is similar to calling a sub-WebReport and has the same syntax as RUNSWR. However, there are some key differences:

  1. The template ignores its own parameters, constants, and data source.

  2. The context from the calling report is shared with the template. That means constants, variables, parameters, and the data source from the calling report are visible in the template.

  3. The template can change variables or override the datasource, for example SETDATASOURCE, but these changes are not reflected in the calling report.

The RUNTEMPLATE sub-tag can also be used to run templates located on the filesystem.

Related Information

For more information, see SETDATASOURCE and RUNSWR.
