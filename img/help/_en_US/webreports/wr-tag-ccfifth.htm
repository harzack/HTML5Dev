This section details the content control tags: IF, INCLUDEDISTINCT, INCLUDEIF, INCLUDERANGE, INSERTJSARRAY and INSERTJSON.

IF

Syntax

[LL_WEBREPORT_IF (expression) [:USECASE] /]

[LL_WEBREPORT_ELSEIF (expression) [:USECASE] /] (optional)

[LL_WEBREPORT_ELSE /] (optional)

[LL_WEBREPORT_ENDIF /]

Description

If a condition is met, include the contents in the output. Anything between the IF and ENDIF tags will only be included if the specified condition evaluates to true. If an optional ELSE clause is included then everything between the ELSE tag and the ENDIF tag are included if the specified condition evaluates to false.

This tag differs from the INCLUDEIF tag in that this structure allows selected parts of the row to be conditionally included or excluded. This tag can be used anywhere in the Reportview. When including or excluding the whole row section always use the INCLUDEIF tag rather than IF as this will give better performance.

Parameters

Format for <expression>:

"parmA" <operator> "parmB" <optional clauses>

Example

[LL_WEBREPORT_IF (expression) /]
[LL_WEBREPORT_ELSE /] (optional)
[LL_WEBREPORT_ENDIF /]

Related Information

See the section on Logical Expressions for a list of Operators. See the Notes sections under EXITIF and INCLUDEIF for more information. For more examples of expressions see the Example.

INCLUDEDISTINCT

Syntax

[LL_WEBREPORT_INCLUDEDISTINCT “<distinct_key>” /]

Description

Used to limit the rows returned by the WebReport to so that only unique key values are returned.

The <distinct_key> value specifies which column, or WebReports data tag, should be unique. <distinct_key> can be a column name, a column number or a WebReports Tag/Sub-tag combination.

Only rows that result in a unique value for this key will be included from the data source. Any rows that result in the same value for the specified key will be discarded.

 

  1. The matching for distinct keys is case insensitive

  2. Where there are rows with duplicate keys, the first row in the current order is kept, and all subsequent rows are discarded.

  3. The order of rows used for duplicate comparison is based on the original data source order and any WebReports SORT tags that are included in the row section.

  4. The filtering of rows using this tag occurs before other row filter tags such as INCLUDEIF, EXITIF and INCLUDERANGE.

 

Example

For example, given the following data set:

DataID
FruitName
12345
Apple
12346
Orange
12347
Apple
12348
Banana
12349
Orange
12350
APPLE

If the row section had an INCLUDEDISTINCT filter, such as [LL_WEBREPORT_INCLUDEDISTINCT "FruitName" /], this would result in only rows that had a unique value in the “FruitName” column, as follows:

DataID
FruitName
12345
Apple
12346
Orange
12348
Banana

The same result would occur if the column number 1 was used, for example, [LL_WEBREPORT_INCLUDEDISTINCT "1" /]. This tag also supports the use of WebReports tag and sub-tag combinations. For example, using the original data example and the [LL_WEBREPORT_INCLUDEDISTINCT "[LL_REPTAG=FruitName DECODE:Banana:apple /]" /] tag, the following rows would remain after the filter was executed:

DataID
FruitName
12345
Apple
12346
Orange

In this example, the WebReports sub-tag DECODE converts any data that matches the string “Banana” to “apple”. As the data value “Apple” already exists in a previous row, the row that normally has the value “Banana” in this column, is no longer unique. This feature is most useful when sub-tags such as NODEINFO and / or CAT are used to translate dataid fields into other related data values that might not be unique.

INCLUDEIF

Syntax

[LL_WEBREPORT_INCLUDEIF <expression> /]

Description

Used to determine whether or not a row of data will be included. If this condition is TRUE, then the whole row will be included, otherwise the whole row will be ignored. It is possible to use multiple INCLUDEIF tags to create a series of conditional gates. Multiple INCLUDEIF tags behave as if the conditions are placed together with ANDs, not OR's. For example, the more INCLUDEIF tags used, the more restricted the report. This tag only works in the row section. For example, between the [LL_WEBREPORT_STARTROW /] and [LL_WEBREPORT_ENDROW /] tags.

 

  1. This tag affects the number of rows that are actually returned and displayed. For this reason, this tag has an impact on the ROWNUM tag which should not be used as part of the conditional logic for INCLUDEIF.

  2. This tag does not change the TOTALROWS tag which will always reflect the number of rows that were actually available from the database table. To access the number of rows which are actually returned, in other words, that pass the conditional test, the ACTUALROWS tag should be used. See the specific syntax information for these tags.

 

Parameters

Format for <expression>: "parmA" <operator> "parmB" <optional clauses>

Example

[LL_WEBREPORT_INCLUDEIF "[LL_REPTAG_&var1 /]" != "[LL_REPTAG=DATAID /]" /]
[LL_WEBREPORT_INCLUDEIF "Admin" == "[LL_REPTAG_USERNAME /]" /]
[LL_WEBREPORT_INCLUDEIF "1234" NOTIN "[LL_REPTAG_1 /]" /]
[LL_WEBREPORT_INCLUDEIF "*DELETED*" IN "[LL_REPTAG=keyfield /]" /]
[LL_WEBREPORT_INCLUDEIF "[LL_REPTAG_1 LENGTH /]" >= "10" AND "[LL_REPTAG=dataid /]" CHILDOF "[LL_REPTAG_&rootid /]" /]
[LL_WEBREPORT_INCLUDEIF "Case-sensitive name comparison" == "[LL_REPTAG=Name /]":USECASE OR "Case-insensitive name comparison" == "[LL_REPTAG=Name /]" /]

Example

Ensure that, when using quotes, you do not nest the same style of quote within a tag. For example:

[LL_WEBREPORT_INCLUDEIF "[LL_REPTAG_"123456" /]"

"[LL_REPTAG=DATAID /]" /] is invalid syntax, because “123456” uses double-quotes, but is contained within a larger tag that is also surrounded by double-quotes: "[LL_REPTAG_"123456" /]"

In this scenario, use a mixture of single and double quotes. The following are both examples of valid syntax:

[LL_WEBREPORT_INCLUDEIF "[LL_REPTAG_'123456' /]" /]
[LL_WEBREPORT_INCLUDEIF '[LL_REPTAG_"123456" /]' /]

Related Information

See the Notes section under EXITIF. See also Logical Expressions and Operators.

INCLUDERANGE

Syntax

[LL_WEBREPORT_INCLUDERANGE /]

[LL_WEBREPORT_INCLUDERANGE STARTROW:<n> ENDROW:<n> MAXROWS:<n> /]

Description

Used to limit the rows returned by the WebReport to a range according to specified parameters.

Parameters

At least one of the optional parameters: STARTROW:<n>, ENDROW:<n> or MAXROWS:<n> must be specified. These are treated as follows:

  • If only STARTROW:<n> is specified, and no MAXROWS:<n> or ENDROW:<n> parameters are included, then ENDROW:<n> is assumed to be the last row of the data set.

  • If no STARTROW:<n> is specified, then the first row in the data set is assumed as the starting point.

  • If MAXROWS:<n> and ENDROW:<n> are both specified, then the more restrictive of the two parameters is used. For example, if STARTROW=1, ENDROW=10, and MAXROWS=5, only five rows will be returned.

  • If the STARTROW:<n> parameter is less than 1, then 1 is assumed.

  • If the STARTROW:<n> parameter is greater than the total number of rows in the data set, then no rows will be returned.

  • If the ENDROW:<n> parameter is less than the STARTROW:<n> parameter, then no rows will be returned.

Example

The range of rows being returned are calculated after the other row filters, for example INCLUDEIF or EXITIF, have been evaluated. Consider the following data set:

Source Row#
Data
1
Apple
2
Orange
3
Pear
4
Banana
5
Strawberry
6
Mango

If the row section had an INCLUDEIF filter, such as [LL_WEBREPORT_INCLUDEIF "[LL_REPTAG_SOURCEROWNUM ODDEVEN /]" == "EVEN" /], this would result in only even row numbers being returned. For example:

Source Row#
Data
2
Orange
4
Banana
6
Mango

If an INCLUDERANGE filter was also used, [LL_WEBREPORT_INCLUDERANGE STARTROW:2 MAXROWS:2 /], then this set of rows is whittled down further. This would result in only 2 rows being returned, starting with row 2 from the data set returned after any INCLUDEIF or EXITIF tags have been processed. Using the previous table examples, this set would be:

Source Row#
Data
4
Banana
6
Mango

INSERTJSARRAY

Syntax

[LL_WEBREPORT_INSERTJSARRAY /]

[LL_WEBREPORT_INSERTJSARRAY ARRAYNAME:<Optional Array Name> MULTIVALUE:<Column Name> /]

Description

Used to automatically insert a JavaScript array with all of the report data arranged in a multi-dimensional array.

This will normally create an array called “repData” which is indexed numerically by row and column. It also creates a set of constants to allow the columns to be referenced. For example, to reference the dataid column for a row of data the following could be used:

document.write(repData[row][DATAID]);

It is also possible to use an alternative array name using the ARRAYNAME: parameter. The MULTIVALUE: parameter can be used when one single column of a table is unique and the other columns are repeated.

Related Information

For additional information about using this tag see JavaScript Arrays.

INSERTJSON

Syntax

[LL_WEBREPORT_INSERTJSON /]

[LL_WEBREPORT_INSERTJSON <multiple directives> /]

Description

Used to automatically insert Content Server data into the output formatted as a JSON (JavaScript Object Notation) structure.

Parameters

This tag supports a variety of “directives” that specify options regarding what data and formats to use.

The available directives are:

@ADDJSVAR
@ALLDATASOURCE
@ALLSTATIC
@BROWSECOLUMNS
@ESCAPETEXT
@EXCLUSIVE
@FIELDS
@ROWCOLUMNS

Related Information

For additional information about these directives see Inserting JSON Data.
