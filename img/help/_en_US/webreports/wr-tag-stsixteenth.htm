SCACTION

Syntax

SCACTION:<parms>

Description

Provides the ability to change Security Clearance data for the nodeID supplied in the main tag. If the Security Clearance module is not installed, this sub-tag returns nothing for all options.

Parameters

Syntax
Description
SCACTION:CURRENTSECURITY:<value>
Sets the Current Security Clearance Level. <value> must be an integer. If <value> is zero or a negative integer then the Current Security Clearance Level will be set to <None>.
SCACTION:SUPPLEMENTALMARKINGS:ADD:<value>
Will add the supplemental marking matching the value to the node.
SCACTION:SUPPLEMENTALMARKINGS:REMOVE
Will remove all supplemental markings from the node.
SCACTION:SUPPLEMENTALMARKINGS:REMOVE:<value>
Will remove the supplemental marking matching the value from the node.

Examples

[LL_REPTAG_'123456' SCACTION:CURRENTSECURITY:1 /] will set the Current Security Clearance Level to 1.

[LL_REPTAG_'123456' SCACTION:CURRENTSECURITY:0 /] will set the Current Security Clearance Level to <None>.

Example: [LL_REPTAG_'123456' SCACTION:SUPPLEMENTALMARKINGS:ADD:"A" /] will add the supplemental marking with code “A” to the node.

Example: [LL_REPTAG_'123456' SCACTION:SUPPLEMENTALMARKINGS:REMOVE:"A" /] will remove the supplemental marking with code “A” from the node.

Related Information

For more information, see SCINFO and SCUSERACTION.

SECURETOKEN

Syntax

SECURETOKEN

or

SECURETOKEN:FULLPARM

Description

This sub-tag will generate a secure request token for a specific action or WebReport. This functionality is only supported in Content Server 10 and higher. The data specified by the data tag is used as part of the key, and can be any string or number, but should match whatever action is expected in Content Server. WebReports expect the dataID of the WebReport being run but most Content Server actions expect a string such as “Action-Copy”.

Adding the optional :FULLPARM parameter will return the name of the Content Server token parameter as it will appear in a URL. By default, the returned token is escaped so that any plus signs, +, do not get stripped during request submission. When using the optional :FULLPARM parameter, the return value can be conversted to a hidden HTML input field using the URLTOPOST sub-tag.

Examples

[LL_REPTAG_"Action-Move" SECURETOKEN /] returns a secure request token for moving a node in Content Server.

Example: [LL_REPTAG_"Action-Move" SECURETOKEN:FULLPARM /] returns the token including the URL parameter name. For example, &secureRequestToken=<token>

Example: [LL_REPTAG_$WR1 SECURETOKEN /] returns a unique token required to execute a WebReport, defined as the “WR1” constant.

Example: [LL_REPTAG_"Action-Move' SECURETOKEN:FULLPARM URLTOPOST /] returns <INPUT TYPE="HIDDEN" NAME="securerequesttoken" VALUE="<token>">

Example: <INPUT TYPE="HIDDEN" NAME="securerequesttoken" VALUE="[LL_REPTAG_'Action-Move' SECURETOKEN /]"> returns <INPUT TYPE="HIDDEN" NAME="securerequesttoken" VALUE="<token>">

SEPARATE

Syntax

SEPARATE:<separator>

or

SEPARATE:<separator>:<input decimal symbol>:<output decimal symbol>

Description

Separates long numbers by inserting a separator character between every three digits.

The default notation uses commas “,” as separators and periods “.”as decimals, but some countries use the opposite. Using the optional decimal symbol parameters allows you to format the data specific to your location.

Parameters

Parameter
Description
Separator
The separator character that you would like to use to split up the data.
Input Decimal Symbol
Optional parameter that allows the user to specify the symbol that the data in the main tag uses as a decimal point. Will default to a period “.” if unspecified.
Output Decimal Symbol
Optional parameter that allows the user to specifiy the symbol that the output should use as a decimal point. Will default to a period “.” if unspecified.

Example

[LL_REPTAG_'1234' SEPARATE:, /] returns 1,234

[LL_REPTAG_'1234.1' SEPARATE:.:.:, /] returns 1.234,1

[LL_REPTAG_'1234.1' ROUND:2 SEPARATE:, /] returns 1,234.10

SETEMAILATTACH

Syntax

SETEMAILATTACH

Description

This sub-tag attaches documents to email destinations. The main tag or sub-tag to the left must contain the dataid of a Content Server document. This sub-tag is destination specific and will only function in combination with the email destination.

 

Note:
Each time this sub-tag is used, a document is attached to the destination email. It would, therefore, be easy to overload this delivery mechanism. OpenText recommends that you use WebReports variables to keep a running total of the file sizes in order to stop attaching documents when a given threshold is exceeded.

 

Examples

[LL_WEBREPORT_IF "[LL_REPTAG=DATAID NODEINFO:SIZE:BYTES ADDVAR:TOT CURRENTVAL /]" < "500000" /]
   [LL_REPTAG=DATAID SETEMAILATTACH /]
[LL_WEBREPORT_ENDIF /]

In this example:

  • The DATAID column is being returned from the data source.

  • NODEINFO:SIZE:BYTES takes the dataid and returns the size of the latest version of the node, in bytes.

  • ADDVAR takes the size, in bytes, for each row, and then adds the new value to the user defined variable TOT.

  • CURRENTVAL is used to get the running total of the calculation, for example, contents of TOT variable.

  • A WebReport IF checks if the total size of all the nodes so far is less than 500,000 bytes, ~0.5MB. If the total size is less than 0.5MB, the document will be attached to the email.

SETFORM

Syntax

SETFORM:<field details>

or

SETFORM:<field details>:NEXTROW

or

SETFORM:SEQ (update mode only)

or

SETFORM:NEXTSEQ (update mode only)

or

SETFORM:NEXTKEY:<column name> (update mode only)

Description

This sub-tag stores the value of the data tag in a form field, after formatting by any previous sub-tags. It is used in conjunction with the Exporting to a WebForm feature. The WebReport destination must be set to Form in order for SETFORM to have any effect. The field name identifies a column in the destination form to receive the data in the data tag. There needs to be a SETFORM sub-tag for each column of data that is to be set in the form. Each SETFORM sub-tag inserts data into the same record in the destination form until one of the following occurs, causing subsequent SETFORM sub-tags to write to the next record in destination form:

  • The NEXTROW directive is used with the SETFORM sub-tag. This causes all subsequent SETFORM sub-tags to write to the next record in the form.

  • The reportview header section ends. This causes all subsequent SETFORM sub-tags, in the row section, to write to the next record in the form.

  • A new reportview row is started. As the row section is repeated for each row of data in the data source, each iteration of the row section also moves to the next record in the destination form.

  • The reportview row section ends. This causes any subsequent SETFORM sub-tags, in the footer, to write to the next record in the form.

  • The reportview footer section ends.

Update Mode Syntax:

In update mode, which is set when form is selected as a destination, if the form uses a submission mechanism, then it is necessary to specify into which form record each set of data should be stored. The following rules govern how the target record is specified:

  • The target record in a form is determined with a sequence number, referred to as SEQ in these guidelines.

  • The SEQ to use for the current batch of SETFORM data can be set by using SETFORM:SEQ in a data tag that contains the required sequence number. For example, [LL_REPTAG_"5" SETFORM:SEQ /] would result in the current set of data, as determined by other SETFORM sub-tags, being written to the form record which has a SEQ value of 5.

  • SETFORM:SEQ does not itself write any data to the form, it merely identifies a form record.

  • A tag containing SETFORM:SEQ must contain a positive integer in the main tag, otherwise the corresponding set of data will not be written to the form destination table.

  • The NEXTSEQ directive can be used if you want to update multiple Form records. The value in the data tag is used to set the next SEQ number to be used by any subsequent SETFORM sub-tags. This directive doesn't actually write any data to the destination form, it is simply used to set the next SEQ to be used. For example, [LL_REPTAG_"10" SETFORM:NEXTSEQ /] would determine that the next set of data will be written to the form record with SEQ number 10.

  • SETFORM:NEXTSEQ does not, itself, write any data to the form, it merely identifies a form record.

  • The NEXTKEY directive can be used to specify that the first record of data to be updated should match a particular key value, other than the SEQ column. If this directive is used, an additional sub-tag parameter is required to specify a field to be used for matching a key. For example, imagine that a form has a field called “fruit” and that the record with SEQ equal to 5 has a value of “apple” stored in the “fruit” field. The tag [LL_REPTAG_"apple" SETFORM:NEXTKEY:fruit /] would effectively select the record with SEQ equal to 5 as the next record to be written to. In other words, the tag has specified that we want to write to the first record we find where the “fruit” field contains a value of “apple”.

  • NEXTKEY currently only supports fields that are not contained in sets.

  • If you are only updating some of the fields on a Form, then the existing data will be retained.

 

Note:
Update Mode Syntax (Revision Mechanism): in update mode, set when form is selected as a destination, if the form uses a revision mechanism then it is only possible to add a new version of the form data.

The NEXTROW, SEQ, NEXTSEQ, and NEXTKEY arguments will have no effect. Just use SETFORM:<field details> to update the data.

 

Parameters

Format
Description
SETFORM:<fieldname>
Populates a standard form field of any type.
SETFORM:<fieldname>:<rownum>
Populates the first field within a multi-value field if just <fieldname> is used. If the optional <rownum> operator is specified, the corresponding value within the multi-value field will be populated. For example, SETFORM:'Address':6 will populate the sixth value of a multi-value field called “Address”.
SETFORM:<setname>:<setnum>:<fieldname>
Populates a standard form field of any type within the specified set and set number. For example, SETFORM:'Documents':2:'Document Number' will populate the “Document Number” field in set 2 of the “Documents” set.
SETFORM:<setname>:<setnum>:<fieldname>:<rownum>
Populates the first field within a multi-value field within the specified set and set number if just <fieldname> is used. If the optional <rownum> operator is specified, the corresponding value within the multi-value field will be populated. For example SETFORM:'Documents':2:'Address':6 will populate the sixth value of a multi-value field called “Address” in set 2 of the “Documents” set.

Examples

[LL_REPTAG_'1' SETFORM:SEQ /]
[LL_REPTAG_'apple' SETFORM:fruit /]
[LL_REPTAG_'3' SETFORM:NEXTSEQ /]
[LL_REPTAG_'banana' SETFORM:fruit /]

This syntax will update two form records. It will change the values in the “fruit” field to “apple” for the record which has a SEQ value of 1, and it will change the value to “banana” for the record which has a SEQ value of 3.

Example

[LL_REPTAG_'1' SETFORM:SEQ /]
[LL_REPTAG_'apple' SETFORM:fruit /]

Example: If a Form contained two fields, “Fruit” and “Country”, and the form record, which has a SEQ value of 1, contains values of “raspberry” and “France” respectively for these fields, the syntax above would set the value of “Fruit” to “apple” but would not set a value for “Country”, so the value will remain as “France” after the update.

Example

[LL_REPTAG_'apple' SETFORM:NEXTKEY:fruit /]
[LL_REPTAG_'banana' SETFORM:fruit:NEXTROW /]
[LL_REPTAG_'orange' SETFORM:fruit /]

Example: This syntax will find the first form record where the “fruit” field contains “apple”, and change the value to “banana”. It will then update the “fruit” field of the next form record and change the value to “orange”.

Example: If the form record containing “apple” had a SEQ value of 5, the NEXTROW directive would increment the SEQ value to 6.

Related Information

SETWFATTACH

Syntax

SETWFATTACH:<parms>

Description

This sub-tag copies or moves the Content Server node returned from the main tag to the attachments folder of a workflow instance. The sub-tag will only function correctly in one of two circumstances:

  • From a WebReport which has its destination set to Workflow. For more information, see Exporting to a Workflow.

  • If WebReports Workflow Extensions is installed, SETWFATTACH can be used from within a WebReports step inside a running workflow instance.

Parameters

Supported options include INHERITPERMS and INHERITATTRS, both of which can be used at the same time.

Format
Description
SETWFATTACH:COPY
Copies the Content Server node to the attachments folder of the workflow being initiated.
SETWFATTACH:COPY:CURRENT
When the node is copied, only the latest version will be in the new node. If this option is applied to a folder, only the latest version of all sub-items will be copied.
SETWFATTACH:COPY:INHERITATTRS:DESTINATION
When the node is copied, workflow categories and attributes are inherited from the workflow attachment folder.
SETWFATTACH:COPY:INHERITATTRS:MERGED
When the node is copied, workflow categories and attributes are retained from the source node and merged with the categories and attributes of the workflow attachment folder
SETWFATTACH:COPY:INHERITATTRS:SOURCE
When the node is copied, workflow categories and attributes are retained from the source node.
SETWFATTACH:COPY:INHERITPERMS
When the node is copied, workflow permissions are inherited from the workflow attachment folder. If this option is not used, the permissions are retained from the original node.
SETWFATTACH:COPY:NEWNAME:<name>
When the node is copied, it will receive the specified name.
SETWFATTACH:COPY:UNIQUEONLY
The node will only be copied to the attachments folder if the existing name is unique.
By default, if a name clash occurs, the node being copied will have a number appended to the name, and the copy will proceed. If this option is specified, the copy fails if the name is already present.
SETWFATTACH:MOVE
Moves a Content Server node to the attachment folder of a workflow being initiated.
SETWFATTACH:MOVE:INHERITATTRS:DESTINATION
When the node is moved, workflow categories and attributes are inherited from the workflow attachment folder.
SETWFATTACH:MOVE:INHERITATTRS:MERGED
When the node is moved, workflow categories and attributes are retained from the source node and merged with the categories and attributes of the workflow attachment folder.
SETWFATTACH:MOVE:INHERITATTRS:SOURCE
When the node is moved, workflow categories and attributes are retained from the source node.
SETWFATTACH:MOVE:INHERITPERMS
When the node is moved, workflow permissions are inherited from the workflow attachment folder. If this option is not used, the permissions are retained from the original node.
SETWFATTACH:MOVE:NEWNAME:<name>
When the node is moved, it will receive the specified name.
SETWFATTACH:MOVE:UNIQUEONLY
The node will only be moved to the attachments folder if the existing name is unique.
By default, if a name clash occurs, the node being moved will have a number appended to the name and the move will proceed. If this option is specified, the move fails if the name is already present.

Examples

An example showing INHERITPERMS and INHERITATTRS used at the same time:

[LL_REPTAG=DATAID SETWFATTACH:MOVE:INHERITPERMS:INHERITATTRS:DESTINATION /]

Related Information

SETWFATTACH is one of a handful of specialized sub-tags that are called Action Sub-tags. Additional information on using this, and other action sub-tags, can be found in Action Sub-tags.

See NODEACTION for moving or copying Content Server content, irrespective of destination.

Additional, related information can be found in NODEINFO.

SETWFATTR

Syntax

SETWFATTR:<fieldname>

Description

This sub-tag stores the value of the tag, after formatting by any previous sub-tags, in a Workflow attribute. The sub-tag will only function correctly in one of two circumstances:

  • From a WebReport which has its destination set to Workflow. For more information, see Exporting to a Workflow.

  • If WebReports Workflow Extensions is installed, SETWFATTR can be used from within a WebReports step inside a running workflow instance.

Examples

[LL_REPTAG_&POSTCODE SETWFATTR:ADDRESS3 /]

In this case we are taking a parameter called POSTCODE and storing it in a Workflow atribute called ADDRESS3 which exists in the Workflow we are initiating.

Example

If the Attribute being populated is a multi-value, the optional MV operator can be used:

[LL_REPTAG_&ORIGIN SETWFATTR:MV:COUNTRY /]

In this case, the value of the parameter ORIGIN is inserted in the first element of the multi-value workflow attribute called COUNTRY.

Example

If we wanted to populate subsequent elements in the same multi-value Attribute we could use the ++ operator, as follows:

[LL_REPTAG_&ORIGIN SETWFATTR:MV++:COUNTRY /]

to populate the second element.

Example

You can use the TODATE:format list sub-tag to format a date string to set the value of a date attribute. For example, the following syntax will set the value of the “Start Date” attribute to “24th April 1988”:

[LL_REPTAG_'24/04/1988' TODATE:'%d/%m/%Y' SETWFATTR:'Start Date' /]

Related Information

If the Attribute being populated is a date, the TODATE:format list operator must be used to convert the date into a format that Content Server can recognise.

SETWFFORM

Syntax

SETWFFORM:<formname>>:<options>

Description

This sub-tag stores the value of the tag, after formatting by any previous sub-tags, in a Content Server WebForm that is attached to a Workflow. The sub-tag will only function correctly in one of two circumstances:

  • From a WebReport which has its destination set to Workflow. For more information, see Exporting to a Workflow.

The behaviour of SETWFFORM is slightly more complex than SETWFATTR as the WebReport can write to any number of forms that are associated with a Workflow. If you are only updating some of the fields on a Form then the existing data will be retained.

Description

SLICE

Syntax

SLICE:<start index>

or

SLICE:<start index>:<end index>

Description

For any string of characters, the SLICE sub-tag returns a slice of the original string based on a specified <start index> and an optional <end index>. This sub-tag is similar to the SUBSTR sub-tag but allows an <end index> to be used instead of a <width>. All indices count starting at one, 1, so the slice will include all characters starting from, and including, the <start index> and conclude with, and including, the <end index>, if one is specified. If <end index> is omitted, or exceeds the total length of the string, then the remainder of the string is used.

Example

These different examples of the SLICE sub-tag show the different results which would be produced, assuming the associated tag contained the string: “The quick brown fox”.

SLICE:5 returns “quick brown fox”.

SLICE:5:9 returns “quick”.

SLICE:5:100 returns “quick brown fox”.

SORT

Syntax

SORT

Description

The SORT sub-tag operates on a List or RecArray and sorts the contents. How the sub-tag is used depends on the input type:

  • Lists: with a list, the SORT sub-tag accepts one optional parameter, DESC to indicate descending, to determine the direction.

  • RecArrays: with a RecArray, the SORT sub-tag accepts 1 to 10 parameters to determine on which columns to sort. The parameters can be a <field name> or <column index>, and can be prefixed with a minus sign, -, to sort that field in descending order.

Example

[LL_REPTAG_"{2,-3,8,4}" SORT:DESC /] returns “{8,4,2,-3}”.

[LL_REPTAG_DATASOURCE SORT:OWNERID:"-NAME" /]

STATUS

Syntax

STATUS:<parm>

Description

Replaces numeric status values returned from the Content Server database with the word value equivalent.

The TASK parameter, STATUS:TASK, would replace the values -2,-1,0,1,2,3 with “Pending”, “In Progress”, “Issues”, and so on. The parameters WORK, WORKAUDIT, SUBWORK and SUBWORKTASK perform the same action for the values returned from these tables.

Parameters

Format
Description

STRIP

Syntax

STRIP:<parm>

Description

All occurrences of each individual character are removed. This function is case sensitive.

Example

STRIP:Lke acting on a data item with the value “Livelink System” results in “ivlin Systm” being inserted into the WebReport output.

SUBSTR

Syntax

SUBSTR:<start index>

or

SUBSTR:<start index>:<width>

Description

For any string of characters, the SUBSTR sub-tag returns a substring of the original string based on a specified <start index> and an optional <width>. This sub-tag is similar to the SLICE sub-tag but allows a <width> to be used instead of an <end index>. All indices count starting at 1 (one) so the substring will include all characters starting from, and including, the <start index> and include the number of characters specified by <width>, if one is specified. If <width> is omitted, or exceeds the total length of the remaining string, then the remainder of the string is used.

Example

These different examples of the SUBSTR sub-tag show the different results that will be produced, assuming the associated tag contained the string: “The quick brown fox”

SUBSTR:5 returns: “quick brown fox”

SUBSTR:5:5 returns: “quick”

SUBSTR:5:100 returns: “quick brown fox”

SUBTRACT

Syntax

SUBTRACT:<value>

Description

Subtracts a value from the main tag. The exact operation depends on the types. The return value of the expression, [LL_REPTAG_'<x>' ADD:<y> /], will depend on the types of <x> and <y>.

Parameters

Type of x
Type of y
Returns
Example
Integer or Real
Integer or Real
x - y
[LL_REPTAG_'5' SUBTRACT:2.2 /] returns “2.8”.
Date
Integer (seconds)
x - y (date)
[LL_REPTAG_'May 2 2011' SUBTRACT:86400 /] returns “Sun May 01 00:00:00 2011”.
Date
Date
x - y (seconds)
[LL_REPTAG_'May 1 2011' SUBTRACT:'May 2 2011' /] returns “-86400”.

Example

For example, to compare the difference between two dates: [LL_REPTAG_'May 01 2011' SUBTRACT:'Jun 20 2008' /] will return: “90288000”. However, this represents the number of seconds between the dates and isn't very user friendly.

You can format with the TIMESINCE sub-tag to make it more readable: [LL_REPTAG_'May 01 2011' SUBTRACT:'Jun 20 2008' TIMESINCE /] will return: “2 years, 10 months”.

Related Information

For more information, see ADD, DIVIDE, MULTIPLY, MODULUS, and TIMESINCE.
