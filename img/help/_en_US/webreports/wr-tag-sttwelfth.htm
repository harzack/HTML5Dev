This section details the sub-tags: NEXT, NODEACTION and NODEINFO.

NEXT

Syntax

NEXT

Description

Return the same column from the next row of data. If this sub-tag is used on the last row of data, a blank string is returned.

NODEACTION

Syntax

NODEACTION:<operation>

Description

This sub-tag offers powerful capabilities to perform various actions against nodes within Content Server and, as such, its effects should be carefully understood before usage. The sub-tag itself behaves like other WebReports sub-tags in that NODEACTION will not allow the user running the report to perform an action, for example a move, that they would not be able to perform via the regular interface.

Optionally, append :INHERITATTRS:SOURCE to inherit the category and attribute data from the source document.

Table 1-14: NODEACTION:COPY

Operation
Description
NODEACTION:COPY:<container>
Expects a dataid from the main tag. Copies this node to the location specified in <container>. Takes all versions and uses the same name.
NODEACTION:COPY:<container>:CURRENT
Same as the previous sub-tag but only takes the latest, current, version of the source document. If the source object is a container, only the latest version of items within the container will be copied.
NODEACTION:COPY:<container>:INHERITATTRS:DESTINATION
Inherits the category and attribute data from the destination location.
NODEACTION:COPY:<container>:INHERITATTRS:MERGED
Merges the category and attribute data from the source node and the destination location.
NODEACTION:COPY:<container>:INHERITATTRS:SOURCE
Retains the source category and attribute data when copied to the destination location.
NODEACTION:COPY:<container>:INHERITPERMS
Inherits the permissions from the destination container.
NODEACTION:COPY:<container>:INHERITRMOFF
Default behavior for the NODEACTION sub-tag is as if the action occurred through the Content Server interface; that is a copied node will inherit RM Classifications from the new parent container. This option can be used to stop the inheritance of the RM Classification. If an existing RM Classification is already applied to the node that is about to be copied then that RM Classification will be retained after the copy event has occurred. This option can be used to override the “Records Management Administration”, “Managed Objects” and “Inherit from” options but cannot be used to force the inheritance of an RM Classification if the appropriate RM settings are not present.
NODEACTION:COPY:<container>:NEWNAME:<name>
Uses the optional NEWNAME parameter to specify a new name for the destination node.
NODEACTION:COPY:<container>:NOCHILDREN
Copies just the specified node without copying its child nodes.
NODEACTION:COPY:<container>:UNIQUEONLY
If the node name is not unique in the destination container the node will not be copied. This option can be used with or without the NEWNAME option.

The COPY parameters above can be used in any combination and in any order. See Table 1-21 below for an example of combining multiple parameters.

Table 1-15: NODEACTION:CREATE

Operation
Description
NODEACTION:CREATE:COLLECTION:<container>:<name>
Creates a new collection in container. The data in the main tag is ignored.
NODEACTION:CREATE:COMPOUNDDOC:<container>:<name>
Creates a new compound document in <container>. The data in the main tag is ignored.
NODEACTION:CREATE:DOC:<container>:<name>
Expects the dataid of a document in the main tag. The content of the latest version of this document is used as the source of a new document which is created in <container>.
NODEACTION:CREATE:FOLDER:<container>:<name>
Creates a new folder in <container>. The data in the main tag is ignored.
NODEACTION:CREATE:GENERATION:<container>:<name>
Creates a generation from the latest version of the node specified in the main tag. Optionally append :VERSIONNUM:<version number> to create a generation from a specific version of the node instead. For example, NODEACTION:CREATE:GENERATION:<container>:<name>:VERSIONNUM:2 will create a generation from version 2 of the node.
 

Note:
If the INHERITATTRS argument is used in addition to VERSIONNUM then only INHERITATTRS:SOURCE is supported. If the VERSIONNUM argument is not used then CREATE:GENERATION will support INHERITATTRS with both the SOURCE or DESTINATION arguments.

 
NODEACTION:CREATE:PROJECT:<container>:<name>
Creates a new project in <container>. The data in the main tag is ignored.
NODEACTION:CREATE:RELEASE
Creates a Release of the Compound Document specified in the main tag. Optionally append a name for the Release. For example, NODEACTION:CREATE:RELEASE will create a Release using the default name: “1.0”, “2.0”.NODEACTION:CREATE:RELEASE:"Release Name" will create a Release called “Release Name”.
NODEACTION:CREATE:REVISION
Creates a revision of the Compound Document specified in the main tag. Optionally append a name for the Revision. For example, NODEACTION:CREATE:REVISION will create a Revision using the default name: “1.0”, “2.0”.NODEACTION:CREATE:REVISION:"Revision Name" will create a Revision called “Revision Name”.
NODEACTION:CREATE:SHORTCUT:<container>:<name>
Creates a shortcut pointing to the node specified in the main tag.
NODEACTION:CREATE:URL:<container>:<name>
Expects a URL in the main tag. The URL must be fully qualified, for example, http://. It also creates a new URL object in <container>.

All NODEACTION:CREATE options, apart from RELEASE and REVISION support INHERITATTRS, INHERITPERMS and UNIQUEONLY.INHERITPERMS behaves the same as in NODEACTION:COPY and NODEACTION:MOVE.INHERITATTRS behaves slightly differently because there is no concept of existing categories, except in the case of a document which is using an existing node to get the content.INHERITATTRS supports SOURCE and DESTINATION options for documents. It only supports DESTINATION for other object types.INHERITATTRS supports SOURCE for NODEACTION:CREATE:GENERATION when the VERSIONNUM argument is used. If not specified, no categories will be applied.MERGE is not supported for NODEACTION:CREATE.UNIQUEONLY will return either:

  • the node ID of the new node if one is created, or

  • the node ID of an existing node with the same name and subtype if one has prevented a new node from being created, or

  • an empty string containing a blank error message stating that the new node couldn't be created if a node with the same name but a different subtype prevented a new node from being created.

The error message can be displayed instead of the empty string by using the ONERROR:LONG tag.

Table 1-20: NODEACTION:MAXVERS

Operation
Description
NODEACTION:MAXVERS:<versions>
Sets the maximum number of versions for a document or a Custom View in the same way as can be set from the object's Specific tab. Content Server provides an opentext.ini setting, VersionLimitDefault, that defines the default maximum number of versions. A node can be set to this value using the optional DEFAULT parameter. For example, [LL_REPTAG=DATAID NODEACTION:MAXVERS:DEFAULT /]. To set a node to have an unlimited number of versions use the NOLIMIT option. For example, [LL_REPTAG=DATAID NODEACTION:MAXVERS:NOLIMIT /]. See also NODEACTION:PURGEVERSIONS.
For more information, see NODEACTION:MOVE.

Table 1-21: NODEACTION:MOVE

Operation
Description
NODEACTION:MOVE:<container>
Expects a dataid from the main tag. Moves this node to the location specified in <container>.
NODEACTION:MOVE:<container>:INHERITATTRS:DESTINATION
Moves the node to <container> and inherits categories and attributes from there.
NODEACTION:MOVE:<container>:INHERITATTRS:MERGED
Moves the node to <container> and merges categories and attributes from the source node and <container>.
NODEACTION:MOVE:<container>:INHERITATTRS:SOURCE
Moves the node to <container> and retains categories and attributes from the source node.
NODEACTION:MOVE:<container>:INHERITPERMS
Moves the node to <container> and inherits permissions from <container>.
NODEACTION:MOVE:<container>:INHERITRMOFF
Default behavior for NODEACTION is as if the action occurred through the Content Server interface. In otther words, a moved node will inherit RM Classifications from the new parent container. This option can be used to stop the inheritance of the RM Classification. If an existing RM Classification is already applied to the node that is about to be moved, then that RM Classification will be retained after the move event has occurred. This option can be used to override the “Records Management Administration”, “Managed Objects”, and “Inherit from” options, but it cannot be used to force the inheritance of an RM Classification if the appropriate RM settings are not present.
NODEACTION:MOVE:<container>:NEWNAME:<name>
Optionally renames the moved node.
NODEACTION:MOVE:<container>:UNIQUEONLY
If the node name is not unique in the destination container, the node will not be moved. This option can be used with or without the NEWNAME option.

The move parameters above can be used in any combination and in any order. For example, [LL_REPTAG=DATAID NODEACTION:MOVE:1234:INHERITATTRS:MERGED:INHERITPERMS:NEWNAME:"my node" /] to combine the source and destination categories, inherit permissions from the destination and rename the moved node to “my node”. In order to attach Content Server items to a workflow being initiated by a WebReport, see SETWFATTACH. See Action Sub-tags for more information.

Table 1-22: NODEACTION:MOVEPROVIDER

Operation
Description
NODEACTION:MOVEPROVIDER:<provider name>:<version number>
Changes the storage provider for the version. Available storage providers can be identified in the Configure Storage Providers administration page. To change a storage provider to the default provider use DEFAULT as the provider name. If no version number is provided, the latest version will be moved to the new provider. If ALL is specified in place of the version number, all versions of the node will be moved to the new provider. Users should carefully consider the performance implications of this sub-tag, as it is not only updating the database, but it is also moving the underlying storage location. This means a single node with 30 versions will result in 30 files being moved, and if these are each one megabyte documents, then there is a considerable overhead on the system. OpenText recommends to do operations like this outside working hours and in a throttled manner.

Table 1-28: NODEACTION:RENAME

Operation
Description
NODEACTION:RENAME:<new name>
Renames the node returned from the main tag. In Content Server 10 multi-language environments, sets the name in the system default metadata language.
NODEACTION:RENAME:<new name>:<language>
In Content Server 10 multi-language environments, renames the node returned from the main tag in the specified metadata language. For example, NODEACTION:RENAME:"nouveau nom":"fr" will change the French name of the node. Note that the <language> parameter is case-sensitive and must correspond to a value in the “Language Code” column of the Configure Multilingual Metadata admin page.

In Content Server 10 multi-language environments, renames the node returned from the main tag in the user's default metadata language. For example, for a user whose default language is set to French, NODEACTION:RENAME:"nouveau nom":USERLANG will change the French name of the node.
NODEACTION:RENAME:<new name1>:<language1>:<new name2>:<language2>
In Content Server 10 multi-language environments, renames the node returned from the main tag in multiple metadata languages. For example, NODEACTION:RENAME:"nouveau nom":"fr":"new name":"en_US" will change the French name of the node to “nouveau nom” and the English (US) name of the node to “new name”.
Note that the <language> parameters are case-sensitive and must correspond to values in the Language Code column of the Configure Multilingual Metadata admin page.

NODEINFO

Syntax

NODEINFO:<format>

Description

This sub-tag will return information about the current object, node, according to the format parameter that has been provided. The format parameter accepts most columns from the DTREE table in the Content Server database, as well as several other parameters unique to WebReports. The main fields are listed here. This sub-tag acts on either a Content Server nodeid or nickname.

Parameters

The ID of the user who has been assigned to this object, usually a task.

Returns a numeric value representing the object's catalog value.
0 = listed, 1 = featured, and 2 = hidden.

The number of children the node has.

Returns a RecArray containing details of the child nodes of the node ID specified in the main tag.
