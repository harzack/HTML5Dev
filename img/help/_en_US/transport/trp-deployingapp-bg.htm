Note:
You must be a Warehouse Administrator to deploy a Workbench.

 

To deploy a Workbench to the target system, you perform the following steps:

  • Download the Transport Box from the source system, and save the file to an external location (for example, a USB stick or network location).

  • Create a Workbench in the target system’s Transport Warehouse.

  • Upload the Transport Box to the target system using the Add Item menu.

  • Unpack the Transport Box to a Workbench on the target system. After you unpack the box, the Workbench that you specified the box be unpacked appears, where you can view the files that were transported.

     

    Note:
    If you have multiple Boxes that contain items you want to deploy to the target system, you can unpack them to the same Workbench and deploy them to Content Server at the same time.

     

  • Review the Workbench to ensure that item dependencies are met for each item.

  • Deploy the Workbench. The Deploy Workbench page shows the number of items in the Workbench and a description for the Workbench, if one was provided when it was created. The Deploy Workbench page also shows a list of the items that will be deployed with the following information for each item: the type, name, the location the item will be created in, and the status of the item (whether the item is ready to deploy or missing dependencies). If an item is missing dependencies, or not ready to deploy for another reason, it will not be created when the Workbench is deployed. If an item being deployed has the same name as an item in the parent container to which it is being added, it will be added as a new version.

Handling Transport Dependencies

Transport scans the system to verify that item dependencies are met. For example, if an item you are transporting has permissions assigned to it or has an assigned Classification, Transport scans Content Server to verify that the permission and Classification exist. If a Workbench is deployed with an assigned user or group that does not exist on the target system, the permissions on the Workbench are removed; if the user or group exists on the target system, permissions on the Workbench are automatically assigned.

When you open an item on the Workbench, the Overview page appears, and contains the following information:

  • Name – the current name of the item

  • Version – the current version number of the item to be deployed

  • Size – the size of the item

  • Created by – the user name that created the item

  • Created date – the date the original item was created

  • Parent – the folder that the original item was located

  • Originating Site Name – the Content Server instance where the original item was located

  • Description – the description of the item, if one was provided when the item was created

If the item you are deploying has dependencies on other items, they are listed on the item’s Overview page. The following information is available for each dependent item: the item type and name, whether the item exists on the target system, and whether the item exists on the current Workbench. If the dependent item exists on the Workbench, a green check mark appears indicating that the item is ready to deploy.

 

Important:
All dependent items must be in the target system to deploy a Workbench. The Deploy button will be disabled until all dependencies are met.

 

If an item has a dependency on more than one item with the same item type and name, it cannot be deployed. If any item is not ready to be deployed, you can use the Edit button to make changes before starting the deployment. If the parent container of an item does not exist on the target system, you can change the parent container to one that does exist.

 

Note:
The dependency columns Exists on this instance and Exists in Workbench, are only visible to Warehouse Administrators. Also, only Warehouse Administrators can edit item dependencies. The Edit button will not be visible unless you are a Warehouse Administrator.
