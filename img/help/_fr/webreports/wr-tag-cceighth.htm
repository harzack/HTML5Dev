Cette section apporte des informations sur les balises de contrôle de contenu : SORT, STARTROW, STARTSCRIPT, SUBWEBREPORT et SUPPRESSERRORLOG.

SORT

Syntaxe

[LL_WEBREPORT_SORT « <Clé de tri>  »:[ASC/DESC] <Clés de tri supplémentaires> [Ordre général] [« USECASE »] <Directives> /]

Description

Utilisée pour trier l'ensemble de données de rapport, sur la base de la colonne spécifiée. Cette balise doit être utilisée dans la section de modèle de ligne, entre les balises STARTROW et ENDROW. La colonne à trier peut être spécifiée soit par son nom, soit par son numéro. Vous pouvez donc utiliser le numéro de la colonne de données, renvoyé de droite à gauche, ou son nom pour effectuer le tri. Si une balise de rapport est utilisée, le rapport peut également être trié en fonction du résultat de la balise et des sous-balises spécifiées.

Paramètres

Paramètre
Description
Clés de tri
Chaque clé est composée d'un numéro/nom de colonne de référence ou d'une combinaison balise/sous-balise WebReports encadrée par des guillemets simples ou doubles ; cet élément est suivi, de manière facultative, de deux points et d'un paramètre d'ordre, comme illustré dans l'exemple de syntaxe.
Ordre général
Cette valeur peut être DESC ou ASC ; la valeur par défaut étant ASC. Les ordres spécifiés pour chaque clé de tri individuelle remplacent l'ordre général.
Directives
Cette valeur peut être l'une des « directives » au format @PREDEFKEY, @PARMNAMES, avec toute information requise associée. Elle est utilisée pour prédéfinir les clés de tri pouvant être associées aux valeurs d'URL simples. Par exemple, &sortcol=col1 peut être utilisé pour déclencher une clé de tri de type [LL_REPTAG=DATAID CAT:cat1:attr1:display /]. Pour plus d'informations, consultez Tri des résultats.

Exemple

[LL_WEBREPORT_SORT "DATAID" /]
[LL_WEBREPORT_SORT "DATAID" "[LL_REPTAG=DATAID NODEINFO:NAME /]" /]
[LL_WEBREPORT_SORT "PARENTID":DESC "[LL_REPTAG=DATAID NODEINFO:NAME /]":ASC /]
[LL_WEBREPORT_SORT "PARENTID" "[LL_REPTAG=DATAID NODEINFO:NAME /]" /]

Dans le premier exemple ci-dessus, le tri est effectué en suivant l'ordre des ID de données numériques.

Dans le deuxième exemple, le tri est effectué sur la base du nom de nœud, même s'ils utilisent tous deux la même colonne de rapport.

Dans le troisième exemple, le tri s'effectue tout d'abord sur la colonne parentID, dans l'ordre décroissant, puis sur le nom dans l'ordre croissant.

Le quatrième exemple est identique au troisième, excepté que la colonne parentID est triée selon l'ordre par défaut, à savoir l'ordre croissant.

Informations associées

La balise SORT prend également en charge une série de directives qui permettent de prédéfinir les clés de tri et de les associer à des paramètres d'URL simple. Voir Tri des résultats pour plus de détails.

STARTROW

Syntaxe

[LL_WEBREPORT_STARTROW /]

Description

Balise utilisée pour marquer le début du modèle de ligne qui sera appliqué à chaque ligne de données, c'est-à-dire le modèle de ligne de rapport. En d'autres termes, tous les éléments de code situés entre cette balise et [LL_WEBREPORT_ENDROW /] seront répétés pour chaque ligne de données dans la source de données.

Utilisation du modèle de ligne de rapport : cette section de l'affichage de rapport, placée entre les balises STARTROW et ENDROW, peut être utilisée pour générer des lignes table HTML répétées ou pour assigner des données de rapport à un tableau JavaScript, pour un affichage et une utilisation ultérieurs des données.

 

Important :
Le code situé entre ces balises sera répété pour chaque ligne de la source de données associée. Si les balises STARTROW et ENDROW sont omises, toutes les sources de données seront ignorées.

 

STARTSCRIPT

Syntaxe

[LL_WEBREPORT_STARTSCRIPT NAME:<nom de script> /]

Description

Définit un bloc Oscript côté serveur, pouvant être appelé par la balise CALL. Chaque bloc de script porte le nom défini par le paramètre <nom de script>. Il est possible, de manière facultative, de déclarer une ou plusieurs fonctions Oscript à l'intérieur du bloc de script. La première fonction transmet deux paramètres. Le premier paramètre est de type dynamique, auquel sont affectées l'Assoc « context » contenant les données du rapport. Le second paramètre est une List Oscript contenant les arguments indiqués dans la balise CALL, le cas échéant. Lorsqu'un bloc de script est appelé, la première fonction Oscript déclarée dans le bloc est exécutée. Les autres fonctions Oscript déclarées dans le même bloc de script peuvent être appelées par cette première fonction.

 

Note :
Le mode script côté serveur doit être activé par un administrateur, au niveau du serveur et pour chaque WebReport, toutes les fois où un nouvel affichage de rapport est ajouté. Les scripts sont rendus inactifs, excepté lorsqu'ils sont activés. Le mode script côté serveur est un outil puissant, qu'il faut toutefois utiliser avec beaucoup de précaution. Avant de s'en servir, il est recommandé aux utilisateurs de lire attentivement l'ensemble de la documentation disponible à ce sujet. Voir Ecriture de script côté serveur pour obtenir une synthèse sur ce thème et consultez le Guide d'utilisation WebReports (format PDF), via la base de connaissances, qui contient une documentation complète et de nombreux exemples.

 

Exemple

[LL_WEBREPORT_STARTSCRIPT NAME:<script name> /]
function String anyName(Dynamic c, List args)
String result
... 
return result
end
[LL_WEBREPORT_ENDSCRIPT /]

SUBWEBREPORT

Syntaxe

[LL_WEBREPORT_SUBWEBREPORT NODEID:<nodeid du sous-rapport WebReport>PARM:<nom parm1>:<valeur parm1>PARM:<nom parm2>:<valeur parm2> ... <options> /]

Description

Utilisée pour appeler un autre WebReport (un « sous-rapport WebReport »), dont les résultats seront renvoyés et insérés dans le rapport parent, à l'endroit où la balise est placée. Les sous-rapports WebReport peuvent être considérés comme des « sous-routines », qui font partie des résultats de rapport globaux. Cette balise est particulièrement utile lorsque vous souhaitez collecter les résultats de plusieurs requêtes différentes, par exemple des requêtes SQL LiveReport, et les présenter dans un rapport unique. Ainsi, il est souvent plus facile d'utiliser plusieurs requêtes SQL simples à la place d'une seule requête complexe. Vous pouvez également préférer combiner les résultats d'une requête SQL avec ceux d'une autre requête de recherche.

  • Spécifiez le sous-rapport WebReport sur la base de son ID de nœud dans le paramètre NODEID. Dans le deuxième exemple ci-dessus, une constante WebReport est utilisée pour spécifier l'ID de nœud du sous-rapport WebReport.

  • De manière facultative, les paramètres qui seraient normalement transmis à un WebReport via l'URL d'appel peuvent être spécifiés à l'aide de la construction PARM:<nom parm>:<valeur parm>. Pour transmettre les paramètres pouvant contenir des espaces ou le signe : (deux points), servez-vous de guillemets simples ou doubles pour délimiter la valeur des paramètres, comme indiqué dans le premier exemple ci-dessus. Les paramètres du rapport LiveReport « InputLabel » peuvent également être transmis à l'aide de PARM. Si des paramètres requis n'ont pas été configurés, les valeurs par défaut sont utilisées, si elles-mêmes ont été définies dans le sous-rapport WebReport.

  • La balise LL_WEBREPORT_SUBWEBREPORT peut être placée n'importe où dans l'affichage de rapport, mais toute utilisation au sein de la section de ligne doit être gérée avec précaution si le parent ou le sous-rapport WebReport risque de renvoyer un ensemble de résultats important, ce qui pourrait ralentir les performances.

  • Lorsque l'action par défaut définie pour un sous-rapport WebReport est l'exportation des résultats du rapport, plutôt que l'envoi de ceux-ci au navigateur, le message de statut standard sera inséré dans le rapport parent et les résultats du sous-rapport WebReport seront exportés.

  • Portée des paramètres et constantes : les constantes et les paramètres de tous les rapports parents sont pris en compte dans un sous-rapport WebReport. Dans le cas où une même constante ou un même paramètre serait utilisé par différents rapports pour contenir des valeurs différentes, la valeur renvoyée sera celle définie en dernier, c'est-à-dire celle définie par le rapport le plus profondément imbriqué.

  • Les sous-rapports WebReport peuvent eux-mêmes appeler un sous-rapport WebReport ; les rapports peuvent être imbriqués sur un maximum de cinq niveaux, bien que ce chiffre puisse être modifié par l'administrateur Content Server.

  • Si la destination du sous-rapport WebReport est un nœud Content Server, le nodeID peut être renvoyé au WebReport appelant, en ajoutant l'argument SHOWNODEID à l'appel SUBWEBREPORT. Par exemple, [LL_WEBREPORT_SUBWEBREPORT NODEID:[LL_REPTAG_$mySubReport /] SHOWNODEID /]

Exemple

[LL_WEBREPORT_SUBWEBREPORT NODEID:68873 PARM:InputLabel1:299 PARM:myStatus:"awaiting review" /]
[LL_WEBREPORT_SUBWEBREPORT NODEID:[LL_REPTAG_$mySubReport /] /]
[LL_WEBREPORT_SUBWEBREPORT NODEID:[LL_REPTAG_$mySubReport /] SHOWNODEID /]

Informations associées

Pour plus d'informations, consultez Appel d'un sous-rapport WebReport.

SUPPRESSERRORLOG

Syntaxe

[LL_WEBREPORT_SUPPRESSERRORLOG /]

Description

Empêche l'affichage des messages d'erreur dans la sortie. Des erreurs peuvent se produire en cas de saisie ou de placement incorrect des balises.
