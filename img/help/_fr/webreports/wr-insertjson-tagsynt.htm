La syntaxe principale de cette balise est la suivante :

[LL_WEBREPORT_INSERTJSON /]

Cette balise peut être utilisée uniquement dans les parties En-tête et Pied de page de l'affichage de rapport. Contrairement à la balise de contrôle de contenu INSERTJSARRAY, lorsque cette balise est utilisée, la section des lignes est traitée normalement. Il n'est pas valide d'utiliser une balise INSERTJSON dans la section des lignes, car elle peut être utilisée pour renvoyer toutes les lignes de données à l'extérieur de la section des lignes.

La balise INSERTJSON prend en charge de nombreuses « directives » qui contrôlent le type de données renvoyées par la balise. Chaque directive est précédée du signe « @ ». Ce format est différent de celui des autres balises de contrôle de contenu, mais il permet un ensemble de fonctionnalités très flexible. Les directives actuellement prises en charge sont répertoriées au tableau ci-dessous.

Options de balise Description
@BROWSECOLUMNS
Cette directive est prise en charge à partir de Livelink version 9.7.1 uniquement. Elle ne nécessite pas d'informations supplémentaires. Elle permet à la balise INSERTJSON de renvoyer un ensemble de données identique à celui normalement utilisé lors de la navigation des recueils Content Server. Elle est utile pour l'émulation des vues de dossier Content Server, en particulier dans Livelink 9.7.1.
 

Notes :

  • Les lignes de données réelles renvoyées peuvent être filtrées par toute balise de ligne conditionnelle WebReports. Par exemple, les balises INCLUDEIF, EXITIF ou INCLUDERANGE.

  • Cette directive et la directive @ALLDATASOURCE sont mutuellement exclusives.

  • Lorsque cette directive est utilisée dans un WebReport, les éléments renvoyés par la source de données doivent être des éléments Content Server valides, qui existeraient normalement dans un recueil Content Server. Par exemple, si la source de données ne renvoie pas de données Dtree, un message d'erreur est produit. De même, si la source de données renvoie un élément tel qu'un Volume, un message d'erreur est également produit, car cet élément n'apparaît pas normalement dans un dossier.

 
@ALLDATASOURCE
Cette directive ne nécessite pas d'informations supplémentaires. Elle permet à la balise INSERTJSON de renvoyer toutes les informations provenant de la source de données en une seule déclaration. Cette directive crée un tableau avec toutes les lignes et toutes les colonnes. Elle fournit également certains champs standard pour prendre en charge la source de données. Ces champs sont les suivants : sourceID, actualRows, filteredRows, totalRows et totalSourceRows.
 

Note :
Pour le moment, le champ filteredRows est renseigné avec un nombre valide uniquement si la balise [LL_REPTAG_FILTEREDROWS /] est incluse dans le WebReport. Si cette balise n'est pas requise, c'est-à-dire si seule la sortie INSERTJSON est utilisée, la balise [LL_REPTAG_FILTEREDROWS /] doit être placée dans un commentaire.

 
Pour plus d'informations sur ces champs, reportez-vous à la syntaxe de la balise de rapport correspondante, par exemple [LL_REPTAG_SOURCEID /]. Notez que cette directive et la directive @BROWSECOLUMNS sont mutuellement exclusives.
@ALLSTATIC
Cette directive renvoie une structure unique contenant toutes les valeurs de balises statiques normalement renvoyées dans des balises WebReports distinctes. Le nom de chaque champ est identique au nom utilisé dans la balise statique correspondante.
@ROWCOLUMNS <columns>
Cette directive est suivie de champs multiples spécifiant les colonnes supplémentaires à ajouter à chaque ligne renvoyée par les colonnes de la source de données ou de la navigation. Chaque champ peut être le nom d'une colonne ou une balise et des sous-balises WebReports. Ces colonnes sont normalement ajoutées à des colonnes existantes établies par la directive @ALLDATASOURCE ou @BROWSECOLUMNS ; cependant, si aucune de ces directives n'a été utilisée, un tableau appelé myRows est ajouté à la structure JSON résultante, contenant uniquement les colonnes spécifiées dans cette directive @ROWCOLUMNS pour chaque ligne de la source de données. Chaque colonne utilise le format suivant : <nom champ JSON> :"<réf données>"
La référence de données est un simple nom de colonne provenant de la source de données ou une combinaison de balises et sous-balises WebReports. Par exemple, @ROWCOLUMNS newColumn:"[LL_REPTAG=DATAID CAT:price:amount1:display /]"
Cet exemple ajoute une colonne appelée « newColumn » au tableau myRows pour chaque ligne de données renvoyée. La syntaxe de la balise est résolue pour chaque ligne ; dans ce cas, en recherchant une valeur de catégorie/d'attribut.
Autre exemple : @ROWCOLUMNS Column2:"modifyDate"
Dans ce second exemple, un nouveau champ de colonne appelé Column2 est ajouté au tableau myRows en utilisant les données renvoyées par la colonne modifyDate de la source de données.
@EXCLUSIVE PARM:<parm name>
Cette directive définit un mode où seules les données produites par la balise INSERTJSON sont renvoyées. Dans ce mode, toute autre sortie de WebReports est omise ; toutefois, l'affichage de rapport est traité normalement pour assurer que tout traitement de variable nécessaire s'effectue. Ce mode est contrôlé par la présence ou l'absence d'un paramètre dans l'URL. Ce paramètre est spécifié au moyen de l'option PARM: . Par exemple, l'option PARM:jsononly spécifie que si la valeur &jsononly est présente dans l'URL, seules les données JSON sont renvoyées. Par défaut, ce paramètre prend la valeur true. S'il est inclus dans l'URL ou s'il est défini explicitement sur true au moyen de l'expression &jsononly=true, le mode exclusif est utilisé. Inversement, si ce paramètre est omis ou défini explicitement sur false au moyen de l'expression &jsononly=false, le mode normal est utilisé. Notez qu'en mode normal, les données JSON restent incluses dans la sortie du WebReport.
@FIELDS
Cette directive est suivie par des champs multiples, entre guillemets, spécifiant un nom et une valeur pour les champs renvoyés par la balise INSERTJSON. Les champs doivent respecter la syntaxe suivante : <nom_champ>:"<valeur_champ>"
La valeur du champ peut être une valeur littérale, mais il s'agit normalement d'une balise WebReports. Par exemple, USERID:"[LL_REPTAG_USERID /]"
Tout champ spécifié par cette directive est ajouté à l'avant de la structure JSON existante, en plus de tout champ existant pour d'autres directives. Ces champs ne sont pas ajoutés au niveau du tableau myRows, mais uniquement à l'objet de niveau supérieur.
@ADDJSVAR VAR:<nom var>
Cette directive ajoute la variable JavaScript et la syntaxe associée pour assigner la structure JSON renvoyée à la variable spécifiée dans <nom var>. Par exemple, @ADDJSVAR VAR:temp ajoute l'expression suivante au début des données JSON : var temp = "json data object"
@ESCAPETEXT TYPE:<escape type>
Cette directive permet de modifier l'échappement de texte par défaut pour tout champ de texte/chaîne utilisé dans la balise INSERTJSON. Les types d'échappement valides sont : standard et escapeurl. Le type standard utilise la spécification JSON standard pour l'échappement de texte. Le type escapeurl utilise le codage URL à décoder au moyen de la fonction JavaScript decodeURI().

 

Exemple

Dans cet exemple, un WebReport utilise une source de données simplifiée renvoyant seulement trois colonnes, parentId, dataId et subType, et deux lignes du DTREE. Etant donné la syntaxe INSERTJSON suivante :

[LL_WEBREPORT_INSERTJSON
@ADDJSVAR VAR:sampleData
@ALLDATASOURCE
@FIELDS SYSTEMDATE:"[LL_REPTAG_DATE /]"
@ROWCOLUMNS objName:"[LL_REPTAG=DATAID NODEINFO:NAME /]" supplier:"[LL_REPTAG=DATAID CAT:VendorData:supplier:DISPLAY /]"
/]

Remarquez que des retours chariot ont été insérés dans l'exemple ci-dessus pour plus de clarté, mais ils ne doivent pas être utilisés dans l'affichage de rapport.

Voici les données renvoyées au client dans la sortie du WebReport :

  • var sampleData={

  • « SYSTEMDATE »:« Oct 08 2008 »,

  • « sourceID »:129293,

  • « actualRows »:2,

  • « filteredRows »:2,

  • « totalRows »:2,

  • « totalSourceRows »:2,

  • « myRows »:[{« objName »:« AAA test document » , « Supplier »:« Widgets Inc. », « parentid »:126758, « dataid » :127200, « subtype »:144}, {« objName »:« BBB test document » , « Supplier »:« Teletech Assets », « parentid » :126758, « dataid »:129514, « subtype »:144}]

  • }

La capture d'écran ci-dessous illustre un extrait de la structure de données JavaScript établie par le client à partir de ces données JSON, comme affichée par un débogueur de script.

  • L'objet est appelé « sampleData », conformément à la directive ADDJSVAR.

  • Un nouveau champ, appelé SYSTEMDATE, a été ajouté à l'objet, conformément à la directive FIELDS.

  • Les champs sourceID, actualRows, filteredRows, totalRows et totalSourceRows ont été créés automatiquement par la directive ALLDATASOURCE.

  • Deux lignes, soit autant de lignes que présentes dans la source de données, sont stockées dans le tableau myRows. Cet exemple montre les trois champs existants dans la source de données simple : parentid, dataid et subtype. Si la source de données avait renvoyé tous les champs de DTREE, toutes ces colonnes seraient incluses dans le tableau de données myRows, conformément à la directive ALLDATASOURCE.

  • Outre les colonnes renvoyées par la source de données, deux nouvelles colonnes : objName et Supplier ont été ajoutées pour chaque ligne de données. Dans cet exemple simpliste, Supplier utilise le résultat de : [LL_REPTAG=DATAID CAT:'Test Category':'Supplier Company':DISPLAY /] de chaque ligne pour définir la valeur de la donnée appelée Supplier dans chaque ligne du tableau. La valeur objName est obtenue simplement de façon similaire au moyen de la sous-balise NODEINFO:NAME.

  • LiveReports permet d'accéder directement à la base de données de Content Server ; l'autorisation de créer des rapports LiveReport est donc presque toujours réservée aux administrateurs. Un WebReport ne permet pas un accès direct à la base de données ; il utilise plutôt des LiveReports existants, des requêtes de recherche ou d'autres sources de données. WebReports peut donc être utilisé en toute sécurité par des utilisateurs commerciaux et des développeurs qui ne sont pas des administrateurs.

  • Les balises STARTROW et ENDROW ne sont pas visibles. Elles ne doivent PAS être ajoutées car l'éditeur les insère une fois l'édition terminée.

  • Chaque adresse est séparée au moyen du symbole défini comme préférence système. L'instruction pour ce champ indique ce symbole, par exemple une virgule ou un point-virgule.
