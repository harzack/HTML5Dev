SCACTION

Syntaxe

SCACTION:<params>

Description

Donne la possibilité de modifier les données Security Clearance pour le nodeID fourni dans la balise principale. Si le module Security Clearance n'est pas installé, cette sous-balise ne renvoie rien pour toutes les options.

Paramètres

Format
Description
SCACTION:CURRENTSECURITY:<valeur>
Définit le niveau d'habilitation de sécurité actuel. La valeur doit être un entier. Si la valeur est zéro ou un entier négatif, le niveau d'habilitation de sécurité actuel sera défini sur <Aucun>.
SCACTION:SUPPLEMENTALMARKINGS:ADD:<valeur>
Ajoutera le marquage supplémentaire correspondant à la valeur au nœud.
SCACTION:SUPPLEMENTALMARKINGS:REMOVE
Supprimera tous les marquages supplémentaires du nœud.
SCACTION:SUPPLEMENTALMARKINGS:REMOVE:<valeur>
Supprimera le marquage supplémentaire correspondant à la valeur du nœud.

Exemples

Exemple: [LL_REPTAG_'123456' SCACTION:SUPPLEMENTALMARKINGS:ADD:"A" /] ajoutera le marquage supplémentaire de code « A » au nœud.

Exemple

[LL_REPTAG_'123456' SCACTION:CURRENTSECURITY:1 /] définira le niveau d'habilitation de sécurité actuel sur 1.

[LL_REPTAG_'123456' SCACTION:CURRENTSECURITY:0 /] définira le niveau d'habilitation de sécurité actuel sur <Aucun>.

Exemple: [LL_REPTAG_'123456' SCACTION:SUPPLEMENTALMARKINGS:REMOVE:"A" /] supprimera le marquage supplémentaire de code « A » du nœud.

Informations associées

Pour en savoir plus, voir SCINFO.

SECURETOKEN

Syntaxe

SECURETOKEN

ou

SECURETOKEN:FULLPARM

Description

Cette sous-balise va générer un jeton de requête sécurisée pour une action spécifique ou un WebReport. Pris en charge uniquement dans les versions Content Server 10 et supérieures. Les données spécifiées par la balise de données sont utilisées comme partie de la clé et peuvent être toute chaîne ou tout nombre, mais doivent correspondre à l'action attendue dans Content Server. Les WebReports attendent l'exécution des dataID du WebReport, mais la plupart des actions Content Server attendent une chaîne comme « Action-Copier ». L'ajout de l'option FULLPARM va renvoyer le nom du paramètre du jeton Content Server tel qu'il apparaîtra dans une URL. Par défaut, le jeton renvoyé est inséré afin que tous les signes plus (+) ne soient pas retirés pendant l'envoi de la requête. L'option NOESCAPE peut être utilisée pour éviter ce processus d'insertion qui n'est généralement utilisé que lorsque le jeton fait partie d'une requête POST. Par exemple, dans une balise HTML <INPUT ou avec la sous-balise URLTOPOST.

Exemples

Exemple

[LL_REPTAG_'Action-Move' SECURETOKEN /] renvoie un jeton de requête sécurisée pour déplacer un nœud dans Content Server.

[LL_REPTAG_'Action-Move' SECURETOKEN:FULLPARM /] renvoie le nom du paramètre du jeton suivi de la valeur. Par exemple, &secureRequestToken=<jeton>

[LL_REPTAG_$WR1 SECURETOKEN /] renvoie un jeton unique requis pour exécuter un WebReport, défini comme la constante « WR1 ».

En utilisant le paramètre facultatif :FULLPARM, la valeur renvoyée peut être convertie en champ d'entrée HTML masqué à l'aide de la sous-balise URLTOPOST :

[LL_REPTAG_"Action-Move' SECURETOKEN:FULLPARM:NOESCAPE URLTOPOST /] renvoie <INPUT TYPE="HIDDEN" NAME="securerequesttoken" VALUE="< jeton>">

<INPUT TYPE="HIDDEN" NAME="securerequesttoken" VALUE="[LL_REPTAG_'Action-Move' SECURETOKEN:NOESCAPE"> renvoie < INPUT TYPE="HIDDEN" NAME="securerequesttoken" VALUE="<jeton>">.

SEPARATE

Syntaxe

SEPARATE:<séparateur>

ou

SEPARATE:<séparateur>:<symbole décimal d'entrée> :<symbole décimal de sortie>

Description

Sépare les nombres longs en insérant un caractère de séparation tous les trois chiffres.

La notation par défaut utilise des virgules pour les séparateurs et des points pour les décimales, ou inversement dans certains pays. L'utilisation des paramètres facultatifs de symbole décimal permet de formater les données propres à votre emplacement.

Paramètres

Paramètre
Description
Séparateur
Le caractère de séparation que vous souhaiteriez utiliser pour séparer les données.
Symbole décimal d'entrée
Paramètre facultatif qui permet à l'utilisateur d'indiquer le symbole que les données de la balise principale utilisent comme point décimal. Sans précision, un point « . » sera utilisé par défaut.
Symbole décimal de sortie
Paramètre facultatif qui permet à l'utilisateur d'indiquer le symbole que la sortie doit utiliser comme point décimal. Sans précision, un point « . » sera utilisé par défaut.

Exemple

[LL_REPTAG_'1234' SEPARATE:, /] renvoie 1,234

[LL_REPTAG_'1234.1' SEPARATE:.:.:, /] renvoie 1.234,1

[LL_REPTAG_'1234.1' ROUND:2 SEPARATE:, /] renvoie 1,234.10

SETEMAILATTACH

Syntaxe

SETEMAILATTACH

Description

Cette sous-balise joint des documents aux destinations e-mail. La balise principale ou la sous-balise à gauche doivent contenir le dataID d'un document Content Server. Cette sous-balise est propre à la destination et ne fonctionnera qu'en association avec la destination e-mail. Notez qu'à chaque fois que cette sous-balise est utilisée, un document est joint à l'e-mail de destination. Il serait donc facile de surcharger le mécanisme d'exécution. OpenText recommande d'utiliser des variables WebReports pour maintenir un total cumulé des tailles de fichiers et pour arrêter de joindre des documents lorsqu'un seuil défini est dépassé.

Exemples

[LL_WEBREPORT_IF "[LL_REPTAG=DATAID NODEINFO:SIZE:BYTES ADDVAR:TOT CURRENTVAL /]" < "500000" /]

[LL_REPTAG=DATAID SETEMAILATTACH /]

[LL_WEBREPORT_ENDIF /]

Dans cet exemple, la colonne DATAID est renvoyée à partir de la source de données. En outre, NODEINFO:SIZE:BYTES traite le dataID et renvoie la taille de la dernière version du nœud en octets. De plus, ADDVAR traite la taille en octets pour chaque ligne et ajoute la nouvelle valeur à la variable TOT définie par l'utilisateur. Pour terminer, CURRENTVAL est utilisé pour obtenir le total cumulé, par exemple, du contenu de la variable TOT. Un WebReport IF vérifie que la taille totale de tous les nœuds jusqu'à présent est inférieure à 500 000 octets, soit environ 0,5 Mo. Si la taille totale est inférieure à 0,5 Mo, le document sera joint à l'e-mail.

SETFORM

Syntaxe

SETFORM:<détails du champ>

ou

SETFORM:<détails du champ>:NEXTROW

ou

SETFORM:SEQ (mode de mise à jour uniquement)

ou

SETFORM:NEXTSEQ (mode de mise à jour uniquement)

ou

SETFORM:NEXTKEY:<nom colonne> (mode de mise à jour uniquement)

Description

Cette sous-balise stocke la valeur de la balise de données dans un champ de formulaire, après mise en forme par toutes les sous-balises précédentes. Elle est utilisée en association avec la fonctionnalité Exportation vers un WebForm. La destination du WebReport doit être définie sur Formulaire pour que SETFORM fonctionne. Le nom de champ identifie une colonne dans le formulaire de destination pour recevoir les données dans la balise de données. Une sous-balise SETFORM est nécessaire pour chaque colonne de données qui doit être définie dans le formulaire. Chaque sous-balise SETFORM insère les données dans le même enregistrement du formulaire de destination jusqu'à ce que l'une des actions suivantes se produise, les sous-balises SETFORM suivantes écrivant alors sur le prochain enregistrement du formulaire de destination :

  • La directive NEXTROW est utilisée avec la sous-balise SETFORM. Les sous-balises SETFORM suivantes écrivent alors sur le prochain enregistrement du formulaire.

  • La section d'en-tête d'affichage de rapport se termine. Toutes les sous-balises SETFORM suivantes, dans la section de ligne, écrivent alors sur le prochain enregistrement du formulaire.

  • Une nouvelle ligne d'affichage de rapport commence. La section de ligne étant répétée pour chaque ligne de données dans la source de données, chaque itération de la section de ligne se déplace également vers le prochain enregistrement du formulaire de destination.

  • La section de ligne d'affichage de rapport se termine. Les sous-balises SETFORM suivantes, dans le pied de page, écrivent alors sur le prochain enregistrement du formulaire.

  • La section de pied de page d'affichage de rapport se termine.

Syntaxe mode de mise à jour :

En mode de mise à jour, défini lorsque le formulaire est sélectionné en tant que destination, il est nécessaire de spécifier l'enregistrement du formulaire dans lequel chaque ensemble de données doit être stocké. Les règles suivantes définissent la manière dont l'enregistrement cible est spécifié :

  • L'enregistrement cible dans un formulaire est déterminé avec un numéro de séquence, appelé SEQ dans ces instructions.

  • Vous pouvez définir le SEQ à utiliser pour le lot actuel de données SETFORM à l'aide de SETFORM:SEQ dans une balise de données qui contient le numéro de séquence requis. Par exemple, [LL_REPTAG_"5" SETFORM:SEQ /] entraînerait l'écriture de l'ensemble actuel de données, comme déterminé par d'autres sous-balises SETFORM, sur l'enregistrement du formulaire qui possède une valeur SEQ égale à 5.

  • SETFORM:SEQ n'écrit pas lui-même de données sur le formulaire, mais il identifie un enregistrement du formulaire.

  • Une balise contenant SETFORM:SEQ doit comprendre un entier positif dans la balise principale, sinon l'ensemble de données correspondant ne sera pas écrit sur la table de destination du formulaire.

  • La directive NEXTSEQ peut être utilisée si vous souhaitez mettre à jour plusieurs enregistrements du formulaire. La valeur dans la balise de données est utilisée pour définir le prochain numéro SEQ devant être utilisé par les sous-balises SETFORM suivantes. Cette directive n'écrit pas de données sur le formulaire de destination. Elle est simplement utilisée pour définir le prochain SEQ à utiliser. Par exemple, [LL_REPTAG_"10" SETFORM:NEXTSEQ /] déterminerait que le prochain ensemble de données sera écrit sur l'enregistrement du formulaire avec le numéro SEQ 10.

  • SETFORM:NEXTSEQ n'écrit pas lui-même de données sur le formulaire, mais il identifie un enregistrement du formulaire.

  • La directive NEXTKEY peut être utilisée pour indiquer que le premier enregistrement de données à mettre à jour doit correspondre à une valeur de clé particulière, différente de celle de la colonne SEQ. Si cette directive est utilisée, un paramètre de sous-balise supplémentaire est requis pour spécifier un champ à utiliser afin de faire correspondre une clé. Par exemple, supposons qu'un formulaire contienne un champ appelé « fruit » et que l'enregistrement avec SEQ égal à 5 donne la valeur « apple » stockée dans le champ « fruit ». La balise [LL_REPTAG_"apple" SETFORM:NEXTKEY:fruit /] sélectionnerait effectivement l'enregistrement avec SEQ égal à 5 comme étant le prochain enregistrement sur lequel écrire. En d'autres termes, la balise a indiqué que nous voulions écrire sur le premier enregistrement trouvé dans lequel le champ « fruit » contient une valeur correspondant à « apple ».

  • Actuellement, NEXTKEY ne prend en charge que les champs qui ne sont pas contenus dans des ensembles.

  • Si vous mettez à jour seulement certains champs d'un formulaire, les données existantes seront alors conservées.

Paramètres

Format
Description
SETFORM:<nom_champ>
Renseigne un champ de formulaire standard de tout type.
SETFORM:<nom_champ>:<num_ligne>
Renseigne le premier champ dans un champ à valeurs multiples si seul le nom de champ est utilisé. Si l'opérateur facultatif num_ligne est spécifié, la valeur correspondante dans le champ à valeurs multiples sera renseignée. Par exemple, SETFORM:'Address':6 renseignera la sixième valeur d'un champ à valeurs multiples appelé « Adresse ».
SETFORM:<nom_ensemble>:<num_ensemble>:<nom_champ>
Renseigne un champ de formulaire standard de tout type dans l'ensemble et le numéro d'ensemble spécifiés. Par exemple, SETFORM:'Documents':2:'Document Number' va renseigner le champ « Numéro du document » dans l'ensemble 2 de l'ensemble « Documents ».
SETFORM:<nom_ensemble>:<num_ensemble>:<nom_champ>:<num_ligne>
Renseigne le premier champ dans un champ à valeurs multiples dans l'ensemble et le numéro d'ensemble spécifiés si seul le nom de champ est utilisé. Si l'opérateur facultatif num_ligne est spécifié, la valeur correspondante dans le champ à valeurs multiples sera renseignée. Par exemple, SETFORM:'Documents':2:'Address':6 renseignera la sixième valeur d'un champ à valeurs multiples appelé « Adresse » dans l'ensemble 2 de l'ensemble « Documents ».

Exemples

Exemple

[LL_REPTAG=ENDDATE SETFORM:NEWDATE /]

Dans cet exemple, nous prenons une colonne de données appelée ENDDATE dans la source de données et nous la stockons dans un champ appelé NEWDATE existant dans le formulaire que nous renseignons. Cette sous-balise prend en charge tous les types Content Server standard. Par conséquent, dans l'exemple ci-dessus, ENDDATE peut être un entier, une chaîne, une date ou un autre type de données. Vous pouvez par exemple l'utiliser avec un fichier CSV en tant que source de données WebReport et pour lire le contenu directement dans un formulaire. De plus, les données source peuvent être une chaîne, mais nous pouvons, par exemple, utiliser TODATE pour convertir une chaîne au format date Content Server avant d'utiliser SETFORM pour insérer les données en tant que type date Content Server. Par exemple, [LL_REPTAG=ENDDATE TODATE:"%Y/%m/%d" SETFORM:NEWDATE /] va traiter une date dans la source de données au format AAAA/MM/JJ et la convertir au format date Content Server adéquat pour écrire sur le formulaire.

Exemple

[LL_REPTAG_'1' SETFORM:SEQ /]

[LL_REPTAG_'apple' SETFORM:fruit /]

[LL_REPTAG_'3' SETFORM:NEXTSEQ /]

[LL_REPTAG_'banana' SETFORM:fruit /]

Cette syntaxe va mettre à jour deux enregistrements du formulaire. Elle va modifier les valeurs dans le champ « fruit » en « apple » pour l'enregistrement dont la valeur SEQ correspond à 1 et en « banana » pour l'enregistrement dont la valeur SEQ correspond à 3.

Exemple

[LL_REPTAG_'1' SETFORM:SEQ /]

[LL_REPTAG_'apple' SETFORM:fruit /]

Si un formulaire contenait deux champs, « fruit » et « Country », et que l'enregistrement du formulaire, dont la valeur SEQ correspond à 1, contenait les valeurs « raspberry » et « France » respectivement pour ces champs, la syntaxe ci-dessus définirait la valeur « fruit » sur « apple », mais ne définirait pas de valeur pour « Pays ». La valeur resterait donc « France » après la mise à jour.

Exemple

[LL_REPTAG_'apple' SETFORM:fruit /]

[LL_REPTAG_'banana' SETFORM:fruit /]

[LL_REPTAG_'orange' SETFORM:fruit /]

Cette syntaxe trouvera le premier enregistrement du formulaire dans lequel le champ « fruit » contient « apple », et modifiera la valeur en « banana ». Elle mettra alors à jour le champ « fruit » du prochain enregistrement du formulaire et modifiera la valeur en « orange ».

Si l'enregistrement du formulaire contenant « apple » avait une valeur SEQ correspondant à 5, la directive NEXTROW incrémenterait la valeur SEQ à 6.

Informations associées

SETFORM est l'une des rares sous-balises spécialisées appelées Sous-balises d'action. Pour plus d'informations concernant son utilisation et l'utilisation d'autres sous-balises d'action, voir Sous-balises d'action.

SETVAR

Syntaxe

SETVAR:<nom_variable>

Description

Cette sous-balise stocke la valeur de la balise, après la mise en forme par toutes les sous-balises précédentes, dans une variable qui peut être référencée à l'aide du nom spécifié. Cette variable pourrait être référencée ailleurs à l'aide d'une balise comme [LL_REPTAG_%nom_variable /]. Lorsque cette sous-balise est utilisée, la sortie de balise associée ne sera pas affichée, à moins que la sous-balise SHOW soit également utilisée.

Exemples

[LL_REPTAG=DATAID SETVAR:ID /]

Cet exemple stockerait les dataID actuelles dans une variable appelée « ID ».

Informations associées

Pour plus d'informations, voir ADDVAR, CONCATVAR et CURRENTVAL.

Pour plus d'informations concernant l'utilisation de SETVAR, des balises de variables et d'autres sous-balises associées à la gestion des variables, voir Utilisation des variables.

SETWFATTACH

Syntaxe

SETWFATTACH:<params>

Description

Cette sous-balise copie ou déplace le nœud Content Server renvoyé par la balise principale vers le dossier Pièces jointes d'une instance de workflow. La sous-balise ne fonctionnera correctement que dans les deux cas suivants :

  • A partir d'un WebReport dont la destination est définie sur Workflow. Pour en savoir plus, voir Exportation vers un workflow.

  • Si WebReports Extensions for Workflow est installé, la sous-balise SETWFATTACH peut être utilisée à partir d'une étape de WebReport dans une instance de workflow en cours d'exécution.

Paramètres

Les options prises en charge incluent INHERITPERMS et INHERITATTRS.

Format
Description
SETWFATTACH:COPY
Copie le nœud Content Server vers le dossier Pièces jointes du workflow lancé.
SETWFATTACH:COPY:CURRENT
Lorsque le nœud est copié, seule la dernière version sera dans le nouveau nœud. Si cette option est appliquée à un dossier, seule la dernière version de tous les sous-éléments sera copiée.
SETWFATTACH:COPY:INHERITATTRS:DESTINATION
Lorsque le nœud est copié, les catégories et attributs de workflow sont hérités du dossier Pièces jointes du workflow.
SETWFATTACH:COPY:INHERITATTRS:MERGED
Lorsque le nœud est copié, les catégories et attributs de workflow sont conservés depuis le nœud source et fusionnés avec les catégories et attributs du dossier Pièces jointes du workflow.
SETWFATTACH:COPY:INHERITATTRS:SOURCE
Lorsque le nœud est copié, les catégories et attributs de workflow sont conservés depuis le nœud source.
SETWFATTACH:COPY:INHERITPERMS
Lorsque le nœud est copié, les autorisations d'accès au workflow sont héritées du dossier de Pièces jointes du workflow. Si cette option n'est pas utilisée, les autorisations sont conservées depuis le nœud d'origine.
SETWFATTACH:COPY:NEWNAME:<nom>
Lorsque le nœud est copié, il recevra le nom spécifié.
SETWFATTACH:COPY:UNIQUEONLY
Le nom sera copié uniquement vers le dossier Pièces jointes si le nom existant est unique. Par défaut, en cas de conflit de nom, un numéro sera ajouté au nom du nœud copié et la copie continuera. Lorsque cette option est spécifiée, la copie échoue si le nom existe déjà.
SETWFATTACH:MOVE
Déplace un nœud Content Server vers le dossier Pièces jointes d'un workflow lancé.
SETWFATTACH:MOVE:INHERITATTRS:DESTINATION
Lorsque le nœud est déplacé, les catégories et attributs de workflow sont hérités du dossier Pièces jointes du workflow.
SETWFATTACH:MOVE:INHERITATTRS:MERGED
Lorsque le nœud est déplacé, les catégories et attributs de workflow sont conservés depuis le nœud source et fusionnés avec les catégories et attributs du dossier Pièces jointes du workflow.
SETWFATTACH:MOVE:INHERITATTRS:SOURCE
Lorsque le nœud est déplacé, les catégories et attributs de workflow sont conservés depuis le nœud source.
SETWFATTACH:MOVE:INHERITPERMS
Lorsque le nœud est déplacé, les autorisations d'accès au workflow sont héritées du dossier Pièces jointes du workflow. Si cette option n'est pas utilisée, les autorisations sont conservées depuis le nœud d'origine.
SETWFATTACH:MOVE:NEWNAME:<nom>
Lorsque le nœud est déplacé, il recevra le nom spécifié.
SETWFATTACH:MOVE:UNIQUEONLY
Le nœud sera déplacé uniquement vers le dossier Pièces jointes si le nom existant est unique. Par défaut, en cas de conflit de nom, un numéro sera ajouté au nom du nœud déplacé et le déplacement continuera. Lorsque cette option est spécifiée, le déplacement échoue si le nom existe déjà.

Exemples

Notez que INHERITPERMS et INHERITATTRS peuvent être utilisées en même temps :

[LL_REPTAG=DATAID SETWFATTACH:MOVE:INHERITPERMS:INHERITATTRS:DESTINATION /]

SETWFATTR

Syntaxe

SETWFATTR:<nom_champ>

Description

Cette sous-balise stocke la valeur de la balise, après mise en forme par toutes les sous-balises précédentes, dans un attribut de workflow. La sous-balise ne fonctionnera correctement que dans les deux cas suivants :

  • A partir d'un WebReport dont la destination est définie sur Workflow. Pour en savoir plus, voir Exportation vers un workflow.

  • Si WebReports Extensions for Workflow est installé, la sous-balise SETWFATTR peut être utilisée à partir d'une étape de WebReport dans une instance de workflow en cours d'exécution.

Exemples

Exemple

[LL_REPTAG_&POSTCODE SETWFATTR:ADDRESS3> /]

Dans ce cas, nous prenons un paramètre appelé POSTCODE et nous le stockons dans un attribut de workflow appelé ADDRESS3 qui existe dans le Wokflow que nous avons lancé.

Exemple

Si nous renseignons un attribut à valeurs multiples, l'opérateur facultatif MV peut être utilisé :

[LL_REPTAG_&ORIGIN SETWFATTR:MV:COUNTRY /]

Dans ce cas, la valeur du paramètre ORIGIN est insérée dans le premier élément de l'attribut de workflow à valeurs multiples appelé COUNTRY.

Exemple

Si nous voulions renseigner les éléments suivants dans le même attribut à valeurs multiples, nous pourrions utiliser l'opérateur ++, comme suit :

[LL_REPTAG_&ORIGIN SETWFATTR:MV++:COUNTRY /]

pour renseigner le second élément.

Informations associées

Si l'attribut que nous renseignons est une date, l'opérateur TODATE:liste de formats doit être utilisé pour convertir la date en un format reconnu par Content Server.

Cette sous-balise stocke la valeur de la balise, après mise en forme par toutes les sous-balises précédentes, dans un WebForm Content Server joint à un workflow. La sous-balise ne fonctionnera correctement que dans les deux cas suivants :

  • A partir d'un WebReport dont la destination est définie sur Workflow. Pour en savoir plus, voir Exportation vers un workflow.

  • Si WebReports Extensions for Workflow est installé, la sous-balise SETWFFORM peut être utilisée à partir d'une étape de WebReport dans une instance de workflow en cours d'exécution.

Son comportement est légèrement plus complexe que SETWFATTR, le WebReport pouvant écrire sur plusieurs formulaires associés à un workflow. Si vous mettez à jour seulement certains champs d'un formulaire, les données existantes seront alors conservées.

SLICE

Syntaxe

SLICE:<index de début>

ou

SLICE:<index de début>:<index de fin>

Description

Pour toutes les chaînes de caractères, renvoie une vue de recherche de la chaîne d'origine selon un index de début indiqué et un index de fin facultatif. Cette sous-balise est similaire à la sous-balise SUBSTR, mais permet d'utiliser un index de fin au lieu d'une largeur. Tous les index démarrent à 1 (un). Ainsi la vue de recherche va inclure tous les caractères qui incluent et démarrent à partir de l'<index de début>, et tous les caractères qui incluent et finissent avec l'<index de fin>, s'il a été spécifié. Si l'<index de fin> est omis, ou s'il dépasse la longueur totale de la chaîne, le reste de la chaîne est alors utilisé.

Exemple

Ces différents exemples de la sous-balise SLICE illustrent les différents résultats qui seraient produits, en supposant que la balise associée contienne la chaîne : « Le renard roux et rapide ».

SLICE:5 renverrait « renard roux et rapide ».

SLICE:5:9 renverrait « rapide ».

SLICE:5:100 renverrait « renard roux et rapide ».
