Pour la plupart des développements WebReports, il n'est pas nécessaire de comprendre les détails du traitement des affichages de rapports. Cependant, ces informations peuvent être utiles pour certaines situations spécifiques avancées. Par exemple, pour déterminer les balises qui peuvent être utilisées en tant que paramètres pour d'autres balises, certaines instructions utiles vous permettront de mieux comprendre ces scénarios.

En général, les sous-balises peuvent être utilisées dans les balises de données, et les balises de données peuvent être utilisées dans les balises de contrôle de contenu. Il y a cependant des cas dans lesquels vous pouvez utiliser divers types de balises de données dans d'autres balises de données. Le succès de ce type d'imbrication dépend de l'ordre dans lequel chaque type de balise est traité.

Exemple

Dans l'exemple suivant, une balise de paramètre, nodefield, remplace le paramètre de sous-balise NODEINFO. La balise de paramètre direction est insérée en tant que paramètre d'ordre pour [LL_WEBREPORT_SORT /] :

[LL_WEBREPORT_SORT "[LL_REPTAG=DATAID NODEINFO:[LL_REPTAG_&nodefield /] /]" [LL_REPTAG_&direction /] /]

Officiellement, seuls les paramètres, les constantes et les balises de nom de colonne sont pris en charge pour l'insertion dans les balises de données. Mais d'autres combinaisons sont possibles en fonction de l'ordre de traitement ci-dessous. Ces balises de données peuvent être substituées dans les composants suivants de l'affichage de rapport :

  • Paramètres des balises de contrôle de contenu. Les balises de contrôle de contenu font référence à toutes les balises avec le préfixe [LL_WEBREPORT_. Un exemple de balise de contrôle de contenu est le paramètre d'ordre de tri d'une balise de tri.

  • Paramètres de sous-balises.

L'ordre dans lequel les types de balises sont traités

La liste suivante décrit l'ordre dans lequel les différents types de balises sont traités. La compréhension de cet ordre de traitement est utile pour déterminer l'utilisation des balises les unes avec les autres. Par exemple, les paramètres transmis sont traités avant les balises de tri. Par conséquent, toute information transmise dans les paramètres peut être utilisée comme partie de la balise de tri.

  1. Toutes les valeurs de balises de paramètre transmises sont insérées du haut vers le bas dans l'affichage de rapport.

  2. Toutes les valeurs de balises de constante sont insérées du haut vers le bas dans l'affichage de rapport.

  3. Toutes les valeurs de balises de nom de colonne sont insérées du haut vers le bas dans l'affichage de rapport.

  4. Toutes les balises de fonction sont interprétées, et marquées pour une insertion ultérieure, du haut vers le bas dans l'affichage de rapport. Les balises @DATA sont insérées à cette étape.

  5. Toutes les valeurs de balises statiques sont insérées du haut vers le bas dans l'affichage de rapport. En d'autres termes, toutes les balises qui renvoient des informations statiques d'environnement Content Server ne provenant pas de la source de données.

  6. L'affichage de rapport est divisé en trois sections. Toutes les balises de données de ligne sont interprétées, pour une insertion ultérieure, dans la section de ligne.

  7. Toutes les variables sont préparées et marquées pour une résolution ultérieure du haut vers le bas dans l'affichage de rapport.

  8. Toutes les balises de tri sont traitées et les données sont manipulées. Les tris sont réalisés dans l'ordre d'affichage de la balise de tri du haut vers le bas dans l'affichage de rapport.

  9. Toutes les valeurs des balises de rapport, les colonnes de données de la source de données, sont insérées du haut vers le bas dans la section de ligne, de la première ligne à la dernière, selon l'ordre de tri.

  10. Toutes les fonctions et les variables sont insérées du haut vers le bas dans l'affichage de rapport. C'est là que les valeurs de la variable actuelle seront résolues.

  11. Toutes les balises conditionnelles if/endif sont résolues du haut vers le bas dans l'affichage de rapport.

  12. Tous les sous-rapports WebReports sont appelés, et la sortie obtenue insérée, du haut vers le bas dans l'affichage de rapport.

La manière dont les balises sont créées et spécifiées dans l'affichage de rapport est validée chaque fois qu'une nouvelle version est ajoutée au WebReport. Cette validation facilite la recherche d'erreurs de syntaxe pouvant exister dans l'affichage de rapport. Cependant, cette validation peut parfois interférer avec des structures de balises plus complexes. Les utilisateurs chevronnés peuvent souhaiter désactiver la validation pour utiliser des scénarios complexes. Cependant, ceci ne doit être fait qu'après un test de base, avec la validation activée, et de préférence sur un système test afin d'éviter toute éventuelle erreur d'exécution. La validation peut être désactivée en utilisant la balise suivante : [LL_WEBREPORT_DEBUGON /]
