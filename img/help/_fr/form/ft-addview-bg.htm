Les modèles de formulaire peuvent avoir plusieurs vues. Une vue est un fichier représentant une autre version du formulaire par défaut basée sur le modèle du formulaire. Vous pouvez créer une vue lorsque vous souhaitez personnaliser un formulaire pour une raison précise. Vous pouvez par exemple créer une vue personnalisée qui masque un champ que vous ne voulez pas utiliser. Après avoir créé une vue personnalisée, vous pouvez l'afficher ou spécifier quels utilisateurs ou groupes reçoivent cette vue par défaut lorsqu'ils accèdent au formulaire.

A l'instar des modèles de formulaire, les vues sont des fichiers (au format HTML ou PDF) que vous ajoutez à Content Server comme tout autre objet.

 

Note :
Vous ne pouvez ajouter des vues à un modèle que si votre installation Content Server comprend le module Web Forms, le module PDF Forms ou le module eForms.

Si vous ajoutez une version à un modèle de formulaire dont les champs sont différents de la version d'origine, vous devez mettre à jour toutes les vues associées au modèle de formulaire afin que celles-ci soient conformes à la nouvelle version.

 

Définition du type de vue

Lorsque vous ajoutez une vue, vous êtes invité à préciser le type de vue si les modules PDF Forms et WebForms sont installés :

Utilisation des vues <Lecture seule> et <Imprimer>

Deux types de vues personnalisées ajoutent des fonctionnalités au module Forms, <Lecture seule> et <Imprimer>.

Une vue <Lecture seule> permet aux utilisateurs d'afficher un formulaire, mais ne leur permet pas d'apporter de modifications. Avant de créer un formulaire, vous devez configurer les autorisations en lecture seule sur le modèle de formulaire sur lequel repose ce formulaire.

L'ajout de la vue <Imprimer> à un modèle de formulaire permet d'ajouter une commande Imprimer le formulaire au menu Fonctions des formulaires créés avec ce modèle. Cette fonction envoie les données du formulaire à une vue que vous avez conçue spécialement pour l'impression.

 

Note :
Lors de l'ajout de ces types de vue à un modèle, vous devez insérer leur nom entre crochets obliques (< >).

 

Spécification de champs optionnels

Content Server renseigne automatiquement certains champs à l'aide des données stockées sur l'utilisateur ayant ouvert le formulaire. Par exemple, pour le champ de formulaire LL_UserLogin, Content Server fournit le nom d'utilisateur Content Server de l'utilisateur actuel à l'ouverture du formulaire.

Les noms suivants sont réservés pour les champs de données utilisateur (tous respectent la casse) :

  • LL_UserLogin : nom de connexion de l'utilisateur

  • LL_UserFirstName : prénom de l'utilisateur

  • LL_UserMiddleName : second prénom de l'utilisateur

  • LL_UserLastName : nom de l'utilisateur

  • LL_UserFullName : nom complet de l'utilisateur (prénom, second prénom, nom)

  • LL_UserTitle : civilité de l'utilisateur

  • LL_UserMailAddress : adresse e-mail de l'utilisateur

  • LL_UserContact : indicatif et numéro de téléphone de l'utilisateur

  • LL_UserGroupName : département d'habitation de l'utilisateur

 

Important :
Content Server envoie toujours dans ces champs réservés les données relatives à l'utilisateur actuel. Pour stocker ces informations dans Content Server, vous devez autoriser la modification ultérieure du formulaire et conserver une liste des utilisateurs ayant modifié les informations. Vérifiez également que le formulaire est conçu de façon à pouvoir stocker les données dans un autre champ en copiant ou en ajoutant les données dans le champ LL_UserLogin. Si vous essayez de stocker les données en créant un champ dans le modèle de formulaire nommé LL_UserLogin, les valeurs de ce champ seront remplacées par le nom d'utilisateur Content Server de l'utilisateur actuel à chaque modification du formulaire.
